# DSEngine CodeBuddy 测试规则

## Description
DSEngine 项目的统一测试规则，供 CodeBuddy 在 `CTest`、引擎单测、可执行验证与回归检查场景下使用，避免“未测即交付”，并确保测试策略与当前 Windows + CMake 主线工程保持一致。

## Rule Governance
- **规则集版本**: `v1.3.0`
- **生效日期**: `2026-04-13`
- **冲突优先级声明**: 本文件受 `AI_DEVELOPMENT_RULES.md` 约束；当测试效率与“闭环验证”冲突时，以“闭环验证”为准。

## Content
### 1. 测试门禁
- **先测后交付**: 任何代码修改必须至少完成一类可验证检查（单测、专项回归、构建或可执行验证）。
- **失败即阻断**: 测试失败禁止宣称完成，必须修复或明确标记阻塞原因。
- **禁止假阳性**: 若测试总数为 0，不得按“通过”处理，必须输出警告并说明原因。
- **默认主线认知**: 当前本地默认测试门禁以 Windows + MSVC 下的 2D 主线稳定性为优先；3D、Editor、Launcher 及外部工具链相关验证按改动范围与功能开关增量开启。

### 2. CTest 规范
- **统一入口**: C++ 测试统一通过 `ctest` 运行，不允许只在 IDE 手工点跑后直接交付。
- **标签分层**: 至少维护 `engine` 与 `unit` 标签；环境相关或链路更重的测试按需归入 `integration`、`smoke` 等标签。
- **输出要求**: 执行测试时使用失败细节输出（如 `--output-on-failure`），便于快速定位。
- **口径一致**: 新增、删除或重命名测试条目时，`CTest` 注册名、测试标签、脚本入口与 Markdown 文档必须保持一致，不得只改其中一处。

### 3. 构建与开关约定
- **真实开关优先**: 测试与构建流程必须优先使用仓库中真实存在的 CMake 选项、脚本参数与 target 名称，禁止在交付或文档中臆造未实现的开关。
- **当前测试开关**: 引擎测试构建默认以 `DSE_BUILD_ENGINE_TESTS` 为准；如脚本层提供别名参数，必须在脚本中已真实实现后方可引用。
- **功能开关感知**: 新增测试必须显式考虑 `DSE_ENABLE_2D`、`DSE_ENABLE_3D`、`DSE_ENABLE_PHYSX`、`DSE_ENABLE_SPINE` 等功能开关，不得让默认主线门禁误包含未开启能力所依赖的测试。
- **结果可读性**: 构建或测试脚本应尽量保留测试统计（Total/Passed/Failed）与关键失败输出，避免只返回“成功/失败”而无定位信息。

### 4. 新增测试要求
- **目录延续现状**: 新增测试优先沿用现有 `tests/engine/`、`tests/modules/`、`tests/spine/` 等结构，按模块或专项归档，禁止为同类测试随意新起一套平行目录。
- **最小覆盖**: 新增或修复核心逻辑时，至少补一条对应回归测试。
- **增量回归优先**: 当前阶段不强制声明统一全局覆盖率红线；对新增或修改的核心路径，必须优先补齐可防回归的增量测试，并在必要时说明尚未覆盖的边界。
- **可重复执行**: 测试不得依赖开发机临时状态、手工残留文件或未签入资源，确保本地可重复执行。
- **资源约束**: 测试若依赖场景、脚本、贴图、字体或其他资产，必须优先使用仓库内已签入资源；禁止依赖个人目录、绝对路径或不可复现的外部文件。
- **轻重分层**: 纯逻辑回归优先实现为轻量单测；依赖 runtime、Lua bootstrap、窗口系统、图形上下文或真实设备的测试，应归入 `smoke` 或专项 gate，避免拖慢默认门禁。
- **命名规范**: `TEST_CASE` 名称应清晰表达场景、动作与预期结果；可采用 `Given_When_Then` 风格，但不强制统一模板化命名。
- **命名长度约束**: `TEST_CASE` 名称建议控制在 80 个字符以内；超过时应压缩为“关键前置条件 + 核心行为 + 结果”三段，避免冗长影响日志可读性。
- **用例注释要求**: 每个 `TEST_CASE` 必须提供简短中文注释或说明文本，明确“测试意图/场景前提/预期行为”，禁止空泛描述。
- **覆盖方向要求**: 单测设计必须覆盖正向测试、边界测试、反向测试三类场景；若某类不适用，需在说明中标明原因。
- **覆盖全面性要求**: 每个业务场景下，正向/边界/反向三类用例都应达到“可防回归”的充分覆盖，至少包含核心路径、关键边界点与主要失败分支，避免仅做示例性断言。
- **场景顺序要求**: 同一测试文件中的用例顺序固定为“正向测试 → 边界测试 → 反向测试”。

### 5. 交付说明要求
- **必须汇报命令**: 交付时给出实际执行过的测试命令。
- **必须汇报结果**: 明确测试通过数量、失败数量与关键失败信息。
- **必须说明范围**: 若仅执行最小验证集合，需明确说明本次已覆盖的模块、未覆盖的层级以及原因。
- **必须引用代码**: 变更说明附带关键文件链接，便于快速审阅。

### 6. 验证执行策略
- **最小必要验证优先**: CodeBuddy 默认只执行与本次改动直接相关的最小验证集合，避免无差别全量编译或全量回归造成时间浪费。
- **禁止完全不验证**: 除非用户明确要求跳过验证，否则不得以“节省时间”为由完全不执行编译或测试。
- **全量验证交由人工触发**: 大规模全量构建或全量回归默认由开发者手动触发，CodeBuddy 在交付时需给出推荐命令。
- **新增测试后的提示义务**: 每次新增测试用例后，CodeBuddy 必须提示开发者手动执行编译脚本与测试命令，确保结果在本机复核。
- **平台基线优先**: 当前仓库若无额外说明，本地验证命令、脚本与路径写法默认以 Windows + PowerShell / bat + MSVC 为基线，不得默认假设 Unix shell 环境。

### 7. 集成测试分层策略
- **分层执行原则**: `unit` 作为默认门禁，`integration` 与 `smoke` 作为环境相关层，按可用环境逐步开启。
- **外部依赖豁免条件**: 当缺少窗口系统、图形上下文、真实音频设备、Node 工具链或对应驱动时，可暂缓执行相关 `integration` / `smoke`，但必须在交付说明中写明原因。
- **替代验证要求**: 暂缓环境相关验证时，至少保留 `unit + 构建 + 可执行冒烟` 中的两类验证，不得降为“仅静态阅读”。
- **回补义务**: 一旦环境满足（本机或 CI 专用节点），必须补跑对应 `integration` / `smoke` 并记录结果，避免长期空转。

### 8. 文档与规则同步要求
- **测试文档同步**: 当最小门禁集合、推荐测试命令、测试标签或专项 gate 发生变化时，必须同步更新 `doc/TESTING_CTEST_GUIDE.md` 与相关测试说明文档。
- **规则不得先于实现**: 规则文件可以表达目标方向，但不得把尚未落地的脚本参数、测试入口或验证能力描述成“当前已支持”。
- **脚本与规则互证**: 若规则中引用 `build_fast_tests.bat`、`build_all.bat` 等脚本能力，脚本实现必须可被仓库当前版本直接检索与验证。

## Version History
- `v1.3.0` (`2026-04-13`): 统一为 CodeBuddy 测试规则口径，并把交付与验证职责改写为 CodeBuddy 当前使用场景下的表达。
- `v1.2.0` (`2026-04-13`): 收紧为“以仓库真实实现为准”的测试规则；移除不现实的统一覆盖率硬指标，补充功能开关、资源约束、Windows 基线与文档同步要求。
- `v1.1.0` (`2026-03-26`): 增加集成测试分层策略，明确外部依赖场景下的暂缓条件与回补义务。
- `v1.0.0` (`2026-03-26`): 增加规则治理段，补充与开发总纲的冲突裁决关系。
