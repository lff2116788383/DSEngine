# DSEngine CodeBuddy 测试规范

## Description
DSEngine 项目的统一测试规范，供 CodeBuddy 在测试设计、代码变更验证、测试执行与结果汇报时使用。目标是保证测试口径简洁、真实、可执行，避免“未测即交付”与“把规划当现状”。

## Content
### 1. 基本原则
- **先验证，后交付**: 任何代码改动都必须配套至少一类真实验证，禁止未验证直接宣称完成。
- **真实优先**: 只允许基于仓库中已存在的测试目标、脚本、标签、命令和依赖进行说明，禁止臆造测试能力。
- **失败即阻断**: 测试失败时不得宣称任务完成，必须修复、回退，或明确说明阻塞原因。
- **结果分级**: 测试结果只能表述为“已通过 / 已失败 / 未执行 / 环境暂缓”，禁止使用模糊措辞。

### 2. 当前测试框架口径
- **单元测试框架**: 当前主线使用 **Google Test**。
- **集成测试策略**: 需要隔离协作边界时，推荐使用 **Google Test + GMock**；但只有在仓库真实接入后，才能写成默认实践。
- **性能测试策略**: 关键路径性能回归推荐使用 **Google Benchmark**；但只有在仓库真实接入并建立基线后，才能写成默认门禁。
- **执行入口**: 批量测试执行统一通过 `ctest` 聚合运行。
- **禁止混淆**: 不得把 `ctest` 表述为测试框架；它只是测试发现与执行入口。

### 3. 分层测试策略
- **单元测试（unit）**: 覆盖核心数学库、数据结构、基础算法、纯逻辑模块，要求快速、稳定、可重复。
- **集成测试（integration）**: 覆盖模块交互、系统协作、运行时装配与边界契约，必要时使用 mock 隔离依赖。
- **冒烟测试（smoke）**: 覆盖运行时主链、Lua 宿主、3D 最小场景、资源导入链等高价值主路径。
- **性能测试（performance）**: 覆盖热点算法与关键路径，默认不纳入日常提交门禁。

### 4. 新增测试要求
- **优先补回归**: 修复 bug 或修改核心逻辑时，优先补对应回归测试。
- **边界清晰**: 单元测试避免真实窗口、图形上下文、外部设备和不可控 IO 依赖。
- **资源可复现**: 测试依赖的场景、脚本、贴图、模型等资源必须来自仓库已签入内容。
- **命名清晰**: 测试名称必须准确表达场景、行为和预期结果，避免含糊命名。
- **文件归档一致**: 新增测试优先沿用现有 [`tests/gtest/`](tests/gtest/)、[`tests/engine/`](tests/engine/)、[`tests/modules/`](tests/modules/) 等目录结构。

### 5. 执行与验证要求
- **最小必要验证**: 默认执行与改动直接相关的最小验证集合，避免无差别全量回归。
- **禁止零验证交付**: 除非用户明确要求跳过，否则不得完全不跑任何构建或测试。
- **输出可定位**: 测试命令应尽量包含 `--output-on-failure` 或等价失败细节输出。
- **Windows 基线**: 默认以 Windows + MSVC + CMake + bat/PowerShell 为本地验证基线。

### 6. 标签与结果口径
- **推荐标签**: 按测试性质使用 `unit`、`integration`、`smoke`、`performance`；按项目维度保留 `engine`、`gtest` 等可检索标签。
- **结果汇报要求**: 交付时必须说明实际执行命令、结果状态、覆盖范围，以及未执行部分的原因。
- **禁止假阳性**: 若测试未运行、总数为 0、目标未生成或环境不满足，均不得表述为“通过”。

### 7. 文档同步要求
- **规则与实现一致**: 测试规范、CMake、脚本、测试标签、README 与协作文档必须保持一致。
- **变更后同步**: 测试框架、默认门禁、命令入口、测试标签、构建开关发生变化时，必须同步更新相关文档。
- **规划与现状分离**: 推荐方向可以写入文档，但必须明确标注为“规划中”或“待接入”，不得伪装成当前能力。
