# DSEngine CodeBuddy 开发规则

## Description
DSEngine 项目的核心 CodeBuddy 开发规则，规定了 C++, Lua, React 混合架构下的工作流、技术栈约束和防幻觉底线。

## Globs
*.*

## Rule Governance
- **规则集版本**: `v1.2.0`
- **生效日期**: `2026-06-15`
- **冲突优先级**:
  1. `AI_DEVELOPMENT_RULES.md`（总纲）
  2. `AI_TESTING_RULES.md`（测试约束）
  3. `AI_DOC_STANDARDS.md`（注释与文档表达）
- **裁决原则**: 出现冲突时按优先级执行；低优先级规则不得否定高优先级硬约束。

## Content
### 1. 核心原则
- **谋定而后动**: 修改前必须使用 CodeBuddy 可用工具全面检索代码库，禁止凭空猜测 API。
- **最小化变更**: 优先修改现有文件，避免不必要的新建和重构。
- **一致性**: 严格遵循现有命名规范、缩进和设计模式（如 C++、Lua、React 的不同惯用法）。
- **闭环验证**: 代码变更后必须通过编译、测试或命令验证，不交付未验证代码。

### 2. 技术栈约束
- **C++ (`engine/`)**: 核心层。要求高性能、内存安全，遵循现代 C++ 规范。
  - **内存管理**: 禁用裸指针（`new`/`delete`）进行所有权管理。优先使用现有的内存池和对象池，常规动态分配必须使用智能指针。
  - **错误处理**: 引擎层默认禁用异常（`-fno-exceptions`）。使用断言（`assert` 或自定义宏）处理致命错误，常规错误返回错误码或 `std::optional`。
- **Lua (`script/` & `samples/`)**: 脚本层。遵循表驱动编程，注意 C++/Lua 交互成本。
- **C++ 应用层 (`apps/`)**: 以 C++ 为主 —— 编辑器 `editor_cpp`(ImGui)、`standalone`(独立运行时)、`runtime`(宿主示例)、`tools`(AssetBuilder)、`android_host`、`web_host`。
- **React/TS (`apps/launcher_tauri/`)**: **仅**启动器前端用 Tauri + React/TS；遵循现代前端规范、Hooks 与严格类型检查。其余 `apps/` 子目录均非前端工程。
- **CMake**: 新增文件、库或 target 时必须同步更新 `CMakeLists.txt`。

### 3. 工作流
1. **分析**: 拆解任务，检索上下文；若仍存在关键歧义，优先通过 CodeBuddy 的提问能力澄清，否则基于现有代码做最小合理假设。
2. **规划**: 复杂任务（>3 步）立即使用 CodeBuddy 的 Todo 能力建立任务列表。
3. **执行**:
   - 修改前必须先读取相关文件，并基于准确上下文做定点编辑，禁止脱离文件真实内容盲改。
   - **防乱码与破坏**: 写入文件时注意保留原有 BOM/编码头，严禁使用低级正则脚本进行无脑的批量代码/注释注入，所有修改必须基于准确的语义理解。
4. **交付**: 验证后输出，回复必须带代码引用、关键文件说明和已执行验证结果。

### 4. 防幻觉
- **禁伪造 API/库**: 必须在代码库或 `depends/` / `package.json` 中确认存在。
  - **第三方库**: 严禁未经用户明确授权擅自引入新的第三方库（如通过 vcpkg、git submodule 等）。
  - **跨语言绑定**: 编写 Lua 脚本时，严禁臆造未经 C++ 导出的 API。必须先检索 C++ 绑定代码确认该 API 已真实导出给 Lua。
- **禁高危命令**: 构建、清理与脚本命令局限于项目工作区。
- **禁虚假完成**: 遇阻时 Todo 保持 `in_progress` 并诚实说明，绝不标记完成。

### 5. 架构保护
- 修改核心系统（`ecs/`、`render/`、`core/`）极其谨慎。
- 遵循现有事件总线和作业系统，禁随意引入新并发或新事件机制。
- 架构变动必须同步更新对应 Markdown 文档。

- `v1.1.0` (`2026-04-13`): 统一为 CodeBuddy 使用口径，移除旧工具动作名，收紧为可执行的通用工作流描述。
