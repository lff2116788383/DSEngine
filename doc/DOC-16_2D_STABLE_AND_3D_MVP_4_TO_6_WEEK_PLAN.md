# DOC-16 2D Stable 与 3D MVP 未来 4~6 周执行计划

本文档用于把 DSEngine 下一阶段工作明确收敛到两个目标：

1. 把 2D 主线收成稳定可展示版本
2. 把 3D 收成最小可验证 MVP 闭环

目标不是继续扩张功能列表，而是把“轻量级 / 高性能 / 现代化”从口号转成：

- 有清晰边界
- 有可执行步骤
- 有每周验收口径
- 有明确的不纳入项

---

## 1. 总体策略

未来 4~6 周，按以下顺序推进：

### 第一阶段：2D 主线收口（约 2 周）

目标：

- 让 2D 成为可明确对外表述的稳定主线
- 继续压实测试、文档、编辑器与运行时的一致口径

### 第二阶段：3D MVP 闭环（约 2 周）

目标：

- 让 3D 从“零散能力点集合”收口为“最小可验证闭环”
- 只保留最小 3D MVP 所需链路，不继续无边界扩展

### 第三阶段：性能与对外口径收束（约 1~2 周）

目标：

- 让“轻量级 / 高性能 / 现代化”至少具备最小证据
- 为后续版本决策提供可量化依据

---

## 2. 第 1 周：2D 稳定性与工程收口

### 2.1 本周目标

把 2D 从“能跑”收成“可信”。

### 2.2 重点任务

#### 任务 1：固定 2D 当前最小常用门禁

建议固定并持续使用以下门禁：

- `engine.unit`
- `engine.lua_runtime`
- `engine.lua_runtime.smoke`
- `engine.resource_injection`
- `engine.cpp_runtime`
- `engine.2d.ui`
- `engine.2d.physics2d`
- `engine.2d.particle`
- `engine.2d.localization`
- `engine.spine`
- `engine.spine.smoke`

#### 任务 2：继续清理测试中的不稳定断言形态

重点范围：

- Lua runtime
- resource injection
- runtime smoke

规则统一为：

- 先求值到局部变量
- 再做 `REQUIRE(...)`
- 必要时补 `INFO(...)`

#### 任务 3：统一 Windows 下 Catch runner 约定

固化当前稳定 runner 结构：

- `applyCommandLine(...)`
- `session.run()`
- `quick_exit(result)`

并避免未来重复引入“测试逻辑跑完但 `session.run()` 不返回”的伪超时。

#### 任务 4：检查 2D 编辑器高频主链

本周只聚焦高频闭环：

- Scene save/load
- Hierarchy
- Inspector
- Play/Edit
- 常见 2D 组件编辑体验

只修最影响闭环的 2~3 个问题，不扩新系统。

### 2.3 本周验收

满足以下条件视为完成：

- 常用 2D gate 稳定通过
- 已知 Windows Lua/Catch 假性卡死问题不再出现
- 测试文档 / 规范已补齐
- 能明确列出 2D 当前主线能力与当前边界

---

## 3. 第 2 周：2D 对外口径与编辑器高频体验

### 3.1 本周目标

让 2D 不只是“技术上稳定”，而是“可展示、可说明”。

### 3.2 重点任务

#### 任务 1：形成一个明确的 2D showcase 场景

建议覆盖：

- UI
- localization
- particle
- physics2d
- spine / animation
- tilemap（若当前稳定）

该场景的作用不是炫技，而是统一对内对外口径：

> 这是当前 2D 主线能力的最小演示样例。

#### 任务 2：小修编辑器高频路径

重点看：

- 组件修改后是否立即正确反映
- 场景保存 / 加载是否存在脏状态问题
- Play/Edit 切换是否污染运行态
- Inspector 是否存在明显不一致行为

#### 任务 3：文档上正式固定“2D 是当前稳定主线”

统一对外表述：

- 当前稳定主线 = 2D
- 3D = MVP 收口中

#### 任务 4：补一轮 2D 回归

确保编辑器和工作流修整不会反向带坏底层门禁。

### 3.3 本周验收

满足以下条件视为完成：

- 存在一个明确的 2D showcase/样例场景
- 文档对 2D 与 3D 的阶段口径清晰
- 编辑器高频路径的显著问题有所减少
- 2D 核心门禁仍稳定通过

---

## 4. 第 3 周：3D MVP 闭环定义

### 4.1 本周目标

停止继续泛化“补 3D”，先明确什么叫 3D MVP 完成。

### 4.2 重点任务

#### 任务 1：固定最小 3D MVP 场景

当前建议只保以下元素：

- Mesh
- Camera3D
- Directional Light
- Shadow
- Skybox

可选但不强求：

- 一个基础材质
- 一个静态模型

#### 任务 2：写清 3D MVP 的完成定义

建议完成定义至少包括：

- 能加载
- 能运行
- 能保存
- 能重新加载
- 有 smoke
- 有 regression
- Inspector 可稳定检视关键 3D 组件

#### 任务 3：清点现有 3D 能力，停止扩张

将当前 3D 能力分为三类：

- MVP 必须保留
- 已有但不纳入当前 MVP
- 暂缓处理

#### 任务 4：稳定构建口径

重点确认：

- `DSE_ENABLE_3D=ON` 时最低可运行构建路径
- PhysX 缺失时的退化路径
- 哪些 3D targets 必须被视为当前稳定目标

### 4.3 本周验收

满足以下条件视为完成：

- 最小 3D MVP 场景定义固定
- MVP 完成标准明确成文
- 不再模糊地用“已有很多 3D 模块”代替“已有 3D 闭环”

---

## 5. 第 4 周：3D MVP 回归与工作流收口

### 5.1 本周目标

让 3D MVP 变成“可以稳定验证”的能力，而不是“理论上存在”。

### 5.2 重点任务

#### 任务 1：构建 3D MVP smoke / regression 组合

至少覆盖：

- Scene load
- Runtime startup
- 关键组件 deserialize
- 最小渲染主链是否进入

#### 任务 2：检查编辑器中关键 3D 组件检视链

本周只看：

- MeshRenderer
- Camera3D
- DirectionalLight
- Skybox

#### 任务 3：检查 3D Scene save/load 最小正确性

重点确认：

- 保存后再加载是否一致
- 关键参数是否丢失
- 路径模型是否稳定

#### 任务 4：约束 MVP 用到的最小材质路径

本周不扩复杂材质系统，只做一件事：

- 把 MVP 当前使用到的材质路径收成稳定最小集

### 5.3 本周验收

满足以下条件视为完成：

- 3D MVP gate 稳定
- save/load 链路可证明闭环成立
- 能诚实地说“已有最小 3D MVP”

---

## 6. 第 5 周：性能证据最小化建设

### 6.1 本周目标

让“轻量级 / 高性能”至少开始具备最小证据，而不只是表述。

### 6.2 重点任务

#### 任务 1：固定最小性能观测指标

##### 2D

- startup time
- scene load time
- draw calls
- frame update time（粗粒度）

##### 3D MVP

- draw calls
- shadow cost（粗粒度）
- scene load time
- frame timing（粗粒度）

#### 任务 2：形成一份 baseline 记录

要求至少包含：

- 当前数值
- 使用场景
- 构建配置
- 机器环境说明

#### 任务 3：确定哪些指标进入长期观察

本周不一定 hard fail，但至少开始记录并比对。

### 6.3 本周验收

满足以下条件视为完成：

- 有一份最小 baseline 记录
- 能用事实而不是形容词描述 2D/3D MVP 当前性能水平

---

## 7. 第 6 周：对外表述与下一阶段决策

### 7.1 本周目标

确定下一阶段到底只选哪个主方向，而不是继续多线并进。

### 7.2 重点任务

#### 任务 1：形成正式能力表述

建议分三层：

##### 稳定主线

- 2D runtime
- UI
- localization
- particle
- physics2d
- spine
- Lua/C++ runtime

##### MVP / Preview

- 3D MVP scene
- 3D 关键组件 Inspector
- 最小 3D smoke / regression

##### 未完成 / 不承诺

- 完整 3D 工作流
- 完整材质系统
- 完整 3D 物理编辑器
- 商业级 AAA 渲染能力

#### 任务 2：只选择一个大方向进入后续阶段

三选一：

1. 2D 稳定版发布准备
2. 3D MVP 深化
3. 编辑器可用性强化

避免三线同时重压。

### 7.3 本周验收

满足以下条件视为完成：

- 有统一对外能力口径
- 有下一阶段单一主方向
- 有清晰舍弃列表，不再“什么都想做”

---

## 8. 未来 4~6 周明确不做什么

以下事项不是没有价值，而是当前阶段不应抢占前排：

- 大而全资源数据库重构
- 大而全 Prefab 系统
- 完整 Shader Graph / 可视化材质编辑器
- 全量高级后处理矩阵
- 高级 3D 粒子生态
- 大规模 3D 地形生态
- AAA 级渲染效果追求
- 大量新增模块

原因是：

> 它们会在当前主线尚未稳定时，把正在形成的 2D Stable 与 3D MVP 再次打散。

---

## 9. 最终执行原则

将未来 4~6 周的原则收敛为一句话：

> 用 2D 建立可信主线，用 3D 建立可信 MVP，用测试与最小性能证据防止项目再次发散。

---

## 10. 当前立即执行建议

若从本文档开始执行，建议立刻按以下顺序启动：

1. 固定并每日使用 2D 常用门禁
2. 继续清理 Lua / runtime / resource injection 相关测试中的不稳定断言
3. 检查 2D 编辑器高频闭环的前 2~3 个问题
4. 整理一个最小 2D showcase 场景
5. 将 3D 当前能力收敛到最小 MVP 场景定义
