# P0.5 / P1 两周执行计划（落地版）

> 适用时间窗口：2026-04-05 ~ 2026-04-16  
> 当前前提：P0 已完成收尾，`dse_engine` / `dse_editor_cpp` / 关键 2D 测试入口均已恢复闭环。  
> 本计划的目标不是重新讨论方向，而是把“接下来两周每天做什么、产出什么、如何验收”直接固定下来。

---

## 1. 执行目标

两周内完成以下两层目标：

### 第一层：P0.5 稳定化（Week 1）

把当前 **可交付** 状态提升为 **可持续开发 / 可复现验证 / 可发布演示** 状态：

- 建立一键构建与回归验证入口
- 建立首套 2D / Editor / Profiler 性能基线
- 清理构建与配置层的已知坑点
- 把 `editor_cpp` 从“能编译、能启动”推进到“可稳定演示和基础编辑”

### 第二层：P1 起步（Week 2）

优先推进 **2D 商业化工程能力** 中最直接能增强工作流的部分：

- 粒子系统的编辑器化与行为增强
- UI 布局能力的编辑器可视化接入
- Localization 与 UI 文本绑定的第一版闭环
- Profiler 的真实指标接入与历史可视化增强

## 3. 周目标拆分

## Week 1：P0.5 稳定化周

### Day 1：一键验证链路固化

#### 目标
- 固化当前已经手工验证通过的构建/测试命令
- 建立统一入口，降低后续回归成本

#### 任务
- 新增或完善脚本（推荐 `script/` 或根目录批处理）用于：
  - 配置 `DSE_BUILD_ENGINE_TESTS=ON`
  - 配置 `DSE_BUILD_EDITOR=ON`
  - 构建 `dse_engine`
  - 构建 `dse_editor_cpp`
  - 执行 `engine.unit`
  - 执行关键 2D smoke
  - 执行 `[profiler]`
- 输出统一结果摘要（成功/失败）

#### 交付物
- `build_p05_validation.bat` 或等效脚本
- 对应使用说明文档

#### 验收标准
- 新人拉代码后可通过单命令完成核心验证
- 任意一步失败时能明确看到失败环节

## 5. 优先级与取舍规则

如果两周内资源不足，按以下顺序保交付：

1. **自动化验证链路**
2. **性能基线**
3. **editor_cpp 基础可用性**

也就是说：**先保工程化与编辑器主工作流，再保单模块体验增强。**

---

## 6. 风险与应对

### 风险 1：editor_cpp 继续膨胀为单文件维护地狱

**应对：**
- 在新增粒子 / UI / Localization / Profiler 面板时，优先把代码拆为独立函数或子模块文件。

### 风险 2：Week 2 新功能反向破坏 P0 稳定性

**应对：**
- 每天结束前跑关键 smoke
- 所有新面板能力先做“可编辑但最小功能”，避免首版过度设计

### 风险 3：性能基线不稳定、难复现

**应对：**
- 先采“相对指标 + 固定场景 + 固定配置”
- 第一版允许是人工记录，不强求自动化压测平台

---

## 7. 两周结束后的预期状态

如果本计划按预期完成，则 DSEngine 会从“P0 功能闭环完成”进入“P1 工程化起步完成”状态，具体表现为：

- 代码层：有持续回归入口，不再完全依赖手工串命令
- 工具层：`editor_cpp` 可以承接粒子 / UI / Localization / Profiler 的第一版工作流
- 文档层：Roadmap、Progress、Guide、执行计划彼此一致
- 组织层：下一阶段不再需要重新争论“先做什么”，而是直接进入 P1 后半段执行

---

## 8. 建议的立即执行顺序

如果现在立刻开工，建议按下面顺序推进，不要打乱：

1. 先做 **Day 1 一键验证链路**
2. 再做 **Day 2 性能基线**
3. 然后做 **Day 4 / Day 5 editor_cpp 可用性增强**
4. 再进入 Week 2 的粒子 / UI / Localization / Profiler 接入

原因很简单：**没有验证链路和基线，后面所有增强都会变得不可控。**
