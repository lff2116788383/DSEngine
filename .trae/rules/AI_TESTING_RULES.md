# DSEngine AI 测试规则

## Description
DSEngine 项目的统一测试规则，规范 CTest、引擎单测、可执行验证与回归检查的执行标准，避免“未测即交付”。

## Globs
*.*

## Rule Governance
- **规则集版本**: `v1.1.0`
- **生效日期**: `2026-03-26`
- **冲突优先级声明**: 本文件受 `AI_DEVELOPMENT_RULES.md` 约束；当测试效率与“闭环验证”冲突时，以“闭环验证”为准。

## Content
### 1. 测试门禁
- **先测后交付**: 任何代码修改必须至少完成一类可验证检查（单测、集成、构建或可执行验证）。
- **失败即阻断**: 测试失败禁止宣称完成，必须修复或明确标记阻塞原因。
- **禁止假阳性**: 若测试总数为 0，不得按“通过”处理，必须输出警告并说明原因。

### 2. CTest 规范
- **统一入口**: C++ 测试统一通过 `ctest` 运行，不允许只在 IDE 手工点跑后直接交付。
- **标签分层**: 至少维护 `engine` 与 `unit` 标签，后续按需扩展 `integration`、`smoke`。
- **输出要求**: 执行测试时使用失败细节输出（如 `--output-on-failure`），便于快速定位。

### 3. build_all.bat 约定
- **测试开关**: 使用 `BUILD_ENGINE_TESTS` / `--with-tests` / `--no-tests` 控制 CTest 流程。
- **可执行验证开关**: 使用 `VERIFY_EXECUTABLES` / `--with-verify-exe` / `--no-verify-exe` 控制示例程序验证。
- **结果可读性**: 保留测试统计（Total/Passed/Failed）与彩色结果（PASS/FAIL/WARN）。

### 4. 新增测试要求
- **目录约束**: 引擎单测放在 `tests/engine/` 下，按模块拆分文件。
- **最小覆盖**: 新增或修复核心逻辑时，至少补一条对应回归测试。
- **覆盖率目标**: 核心模块（`engine/`、`modules/`）行覆盖率目标不低于 70%；未达标时需在交付说明中给出差距与补测计划。
- **可重复执行**: 测试不得依赖本机临时状态，确保本地与 CI 结果一致。
- **命名规范**: `TEST_CASE` 名称统一采用 `Given_When_Then` 结构，表达前置条件、触发行为与预期结果。
- **命名长度约束**: `TEST_CASE` 名称建议控制在 80 个字符以内；超过时应压缩为“关键前置条件 + 核心行为 + 结果”三段，避免冗长影响日志可读性。
- **用例注释要求**: 每个 `TEST_CASE` 必须提供简短中文注释或说明文本，明确“测试意图/场景前提/预期行为”，禁止空泛描述。
- **覆盖方向要求**: 单测设计必须覆盖正向测试、边界测试、反向测试三类场景；若某类不适用，需在说明中标明原因。
- **覆盖全面性要求**: 每个业务场景下，正向/边界/反向三类用例都应达到“可防回归”的充分覆盖，至少包含核心路径、关键边界点与主要失败分支，避免仅做示例性断言。
- **场景顺序要求**: 同一测试文件中的用例顺序固定为“正向测试 → 边界测试 → 反向测试”。

### 5. 交付说明要求
- **必须汇报命令**: 交付时给出实际执行过的测试命令。
- **必须汇报结果**: 明确测试通过数量、失败数量与关键失败信息。
- **必须引用代码**: 变更说明附带关键文件链接，便于快速审阅。

### 6. 验证执行策略
- **最小必要验证优先**: AI 默认只执行与本次改动直接相关的最小验证集合，避免无差别全量编译/全量回归造成时间浪费。
- **禁止完全不验证**: 除非用户明确要求跳过验证，否则不得以“节省时间”为由完全不执行编译或测试。
- **全量验证交由人工触发**: 大规模全量构建/全量回归默认由开发者手动触发，AI 在交付时需给出推荐命令。
- **新增测试后的提示义务**: 每次新增测试用例后，AI 必须提示开发者手动执行编译脚本与测试命令，确保结果在本机复核。

### 7. 集成测试分层策略
- **分层执行原则**: `unit` 作为默认门禁，`integration` 与 `smoke` 作为环境相关层，按可用环境逐步开启。
- **外部依赖豁免条件**: 当缺少窗口系统、图形上下文、真实音频设备或对应驱动时，可暂缓执行 `integration`，但必须在交付说明中写明原因。
- **替代验证要求**: 暂缓 `integration` 时，至少保留 `unit + 构建 + 可执行冒烟` 中的两类验证，不得降为“仅静态阅读”。
- **回补义务**: 一旦环境满足（本机或 CI 专用节点），必须补跑 `integration` 并记录结果，避免长期空转。
