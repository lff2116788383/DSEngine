# 光子映射：用"扔光子"的方式模拟光线，让渲染更真实

> 一篇写给小白的全局光照入门指南

---

## 一、先从一个问题说起

我们先做一个小实验：

> 现在是晚上，你走进一个房间，打开一盏台灯。灯照亮了书桌，但**天花板角落、床底下、窗帘背面**——这些地方并没有被台灯直接照到，却依然依稀可见。为什么？

答案是：**光的间接传播**（间接光照）。

光线从台灯发出 → 照到桌面 → 一部分被吸收 → 另一部分**反弹**到天花板 → 再从天花板反弹到角落……**光会多次反弹**，照亮那些没有"直接看到"光源的地方。

> **传统渲染**（实时渲染，比如游戏引擎）：只计算"直接光照"——光➡️物体➡️眼睛。光线只弹一次。
>
> **光子映射**：连"间接光照"也算——光➡️物体➡️物体➡️物体➡️眼睛。光线弹很多次。

**结果上的区别：**

| 渲染方式 | 天花板角落 | 床底下 | 焦散效果（水面上的光斑） |
|---------|-----------|--------|---------------------|
| 传统渲染 | ❌ 全黑 | ❌ 全黑 | ❌ 看不到 |
| 光子映射 | ✅ 隐约可见 | ✅ 微微亮 | ✅ 漂亮的光斑 |

## 二、一句话说清光子映射

> **光子映射 = 先向场景里"扔"大量光子，记录它们去了哪里 → 再从相机"看"场景时，用这些记录算间接光照。**

整个过程分两步：

```
第一步：光子追踪（从光源出发）
  光源 ──→ 发射光子 ──→ 光子反弹 ──→ 记录光子落地位置 → 存到"光子图"

第二步：视线追踪（从相机出发）
  相机 ──→ 发射光线 ──→ 碰到物体 ──→ 查找附近的光子 → 计算间接光照
```

听起来像什么？

> 像**警察办案**：先在犯罪现场撒满荧光粉（光子）→ 然后开紫光灯看（视线）→ 脚印用荧光粉标记出来了（间接光照信息）！

## 三、用故事理解核心思想

### 🧪 黑暗房间里的萤火虫

想象你走进一个**完全黑暗的房间**，手里有一瓶萤火虫。

**第一步（光子追踪）：**
1. 你把萤火虫**全部倒出来**（发射光子）
2. 萤火虫在房间里乱飞，撞到墙壁就**反弹**
3. 有的飞累了**落在地上**（光子命中漫反射表面）
4. 你拿出地图，**标记每个萤火虫落下的位置**（构建光子图）
5. 你标记了两种萤火虫：
   - 🔵 **普通萤火虫**（全局光子）—— 所有落地的萤火虫
   - 🔴 **特殊萤火虫**（焦散光子）—— 经过镜子/玻璃反弹后才落地的萤火虫

**第二步（视线追踪 + 估算）：**
1. 现在你站在房间中央，**拿着一根激光笔**（相机光线）
2. 激光笔射到某个位置，你问："这个位置附近有多少萤火虫？"
3. 头上有 5 只蓝色萤火虫 → 说明有间接光照
4. 脚边有 3 只红色萤火虫 → 说明之前有光线经过玻璃聚焦到这里（焦散效果）
5. 附近萤火虫**越多越亮**，**越少越暗**

**大功告成！** 你现在能"看到"这个完全黑暗的房间里每个角落的亮度了。

## 四、光子映射的整体架构

这个系统就像一家**三层的餐厅**：

```
┌─────────────────────────────────────────┐
│  第三层：点单厅（渲染调度层）              │
│  - 接待员（Scene）加载菜单（OBJ模型）       │
│  - 传菜员（Integrator）把控出品流程         │
│  - 厨房助手（Embree）快速切菜（光线求交）   │
├─────────────────────────────────────────┤
│  第二层：核心厨房（核心算法层）              │
│  - 主厨（PhotonMap）管着光子库存            │
│  - 储物架（KD-Tree）让找东西超快            │
│  - 配菜师（BxDF）调出不同材质的"味道"       │
├─────────────────────────────────────────┤
│  第一层：仓库（基础数据层）                  │
│  - 光子筐（Photon struct）                 │
│  - 光线盒（Ray struct）                    │
│  - 食材卡（Primitive/Triangle/BxDF）       │
└─────────────────────────────────────────┘
```

### 各角色分工

| 角色 | 类名 | 干啥的 | 举个🌰 |
|------|------|--------|--------|
| 接待员 | `Scene` | 加载3D模型，搭建场景 | 把OBJ文件变成3D场景 |
| 传菜员 | `Integrator` | 协调渲染流程 | 决定"先做哪步再做哪步" |
| 主厨 | `PhotonMap` | 管理所有光子 | 记录每个光子在哪、能量多少 |
| 储物架 | `KD-Tree` | 快速找到附近光子 | 像图书馆索引卡 |
| 配菜师 | `BxDF` | 模拟材质光学特性 | 木头怎么反光、玻璃怎么折射 |
| 厨房助手 | `Embree` | 快速算"光线撞到哪了" | Intel 出品的高性能光线求交库 |

## 五、关键技术逐一说人话

### 5.1 光子（Photon）—— 渲染世界的 "萤火虫"

一个光子不是真光子，它是一个**数据结构**，记录了三条信息：

```
Photon {
  position:     (x, y, z)   // 落点位置
  wi:           (dx, dy, dz) // 入射方向（从哪飞来的）
  throughput:   (r, g, b)   // 携带的能量（颜色 + 亮度）
}
```

> 就像你记录每只萤火虫：落在哪、从哪飞来的、有多亮。

### 5.2 KD-Tree —— "最强大脑"帮光子找邻居

想象一下：你有 100 万只萤火虫的落地位置，现在你想知道"桌子角附近有哪些萤火虫"，难道要一个一个数吗？

**KD-Tree 就是给光子建了个索引**：

```
无索引查询： 遍历100万个光子，看每个离桌子角多远 → O(n) 太慢了
用KD-Tree：  像翻目录一样，先排除左半区 → 排除右半区 → 定位到小范围 → O(log n) 飞快
```

**具体原理（不需要懂代码）：**

| 步骤 | 怎么做 | 就像 |
|------|--------|------|
| ① 排序 | 把所有光子按 X 坐标排序，取中间那个当"根" | 把一堆文件按字母排序，取 M |
| ② 分割 | 左边的光子归左子树，右边的归右子树 | A-L 放左边抽屉，N-Z 放右边 |
| ③ 再分割 | 下一层用 Y 坐标分割，再下一层用 Z 坐标 | 左边抽屉再按第二位字母细分 |
| ④ 递归 | 直到每个节点只包含几个光子 | 每个抽屉只有 10 个文件 |

**查询时：**
- 我要找"桌子角附近的光子" → 先确定桌子角在空间的哪一侧
- 直接忽略掉另一侧的所有光子 → 省了 50% 的查找工作

### 5.3 为什么需要两张光子图？

系统维护了 **两种光子图**：

| 光子图 | 标记 | 条件 | 用途 |
|--------|------|------|------|
| 🔵 全局光子图 | 所有落地的萤火虫 | 只要光子撞到漫反射表面就记录 | 计算普通的间接光照 |
| 🔴 焦散光子图 | 经过镜面反射才落地的 | 必须经过镜子/玻璃 → 再撞到漫反射 | 计算焦散效果 |

**为什么分开？**

因为焦散效果（比如阳光穿过玻璃杯在地面形成的光斑）非常亮、范围小，如果混在全局光子里，需要极大量光子才能清晰呈现。**分开存储 = 给焦散"特殊照顾"**，用更少的光子算出更清晰的效果。

## 六、渲染流程：一步步看清全过程

### 第 1 步：光子追踪

```
光源
  │
  ├── 发射 100 万个光子（并行，使用 OpenMP 多核计算）
  │
  ├── 光子飞出去，遇到物体
  │     ├── 遇到木头（漫反射）→ 记录到全局光子图
  │     │                        → 可能继续反弹（能量变弱）
  │     │
  │     ├── 遇到镜子（镜面反射）→ 继续飞
  │     │   └── 再遇到木头      → 记录到焦散光子图
  │     │
  │     └── 遇到玻璃（折射+反射）→ 一部分反射、一部分穿过去
  │         └── 再遇到木头      → 记录到焦散光子图
  │
  └── 什么时候光子"废了"？（俄罗斯轮盘赌）
        - 每次反弹后，抛硬币
        - 正面 → 继续（能量除以概率）
        - 反面 → 扔掉这个光子
  
  ▶ 最终：两个光子图建好了
```

### 第 2 步：视线追踪（渲染图片）

```
相机（每个像素发射一条光线）
  │
  ├── 光线飞到场景中，碰到物体
  │
  ├── 计算"直接光照"（NEE 直接光源采样）
  │     ├── 从交点向光源发一条"阴影射线"
  │     ├── 没被挡住 → 加上光源的直接贡献
  │     └── 被挡住 → 跳过（阴影）
  │
  ├── 计算"焦散光照"
  │     ├── 在焦散光子图里找最近的 N 个光子
  │     ├── 光子多 → 亮
  │     └── 光子少 → 暗
  │
  └── 计算"间接光照"（最终聚集 Final Gathering）
        ├── 从交点向四面八方发很多条"探测光线"
        ├── 每条探测光线碰到物体后
        ├── 在全局光子图里找附近的光子
        │     └── 附近光子多 → 这条探测光线贡献大
        └── 所有探测光线的结果平均 → 就是间接光照
  
  ▶ 最终：每个像素的颜色 = 直接光照 + 焦散光照 + 间接光照
```

### 为什么需要"最终聚集"？

> **直接查光子图** = 用附近光子的平均亮度 ≈ 会看到"一块一块"的光斑（噪声大）
>
> **最终聚集** = 从交点发很多探测光线查光子图 = 去掉噪声，得到平滑的结果

就像：
- **直接查光子图** ≈ 用 10×10 的马赛克显示图片
- **最终聚集** ≈ 用 1920×1080 的屏幕显示图片

## 七、三种材质的光学模拟

系统支持三种材质，用 BxDF（双向散射分布函数）统一建模：

### 🪵 Lambert 漫反射（木头/墙壁/布料）

| 属性 | 说明 |
|------|------|
| 入射光 | 从任何方向来的光 |
| 反射光 | **均匀地向所有方向反射** |
| 参数 | 只有一个：反射率（颜色） |
| 🌰 | 一张白纸，无论从哪个角度看，亮度都一样 |

**和现实的关系：** 这就是为什么墙壁从侧面看和正面看一样亮。

### 🪞 Mirror 镜面反射（镜子/抛光金属）

| 属性 | 说明 |
|------|------|
| 入射光 | 从一个方向来 |
| 反射光 | **精确地朝一个方向反射**（入射角=反射角） |
| 参数 | 无（完美镜面） |
| 🌰 | 照镜子，你动一下头，看到的完全不同 |

### 🥃 Glass 玻璃（折射+反射）

| 属性 | 说明 |
|------|------|
| 入射光 | 一部分反射、一部分穿透 |
| 反射 vs 折射比例 | 由**菲涅耳效应**决定（Schlick近似） |
| 啥时候全反射 | 从水里看空气时（全内反射） |
| 🌰 | 站在水池边看水面 → 能同时看到倒影（反射）和水底（折射） |

**菲涅耳效应说人话：**

> 你站在湖边，低头看脚下水面 → 几乎透明，能看到水底的鱼（折射为主）
>
> 你看向远处的水面 → 水面变成了镜子，反射天空的云彩（反射为主）

**光子映射精确模拟了这个效应！**

## 八、性能优化：让计算跑得更快

光子映射**非常慢**，因为每个像素要发多条光线，每条光线要查光子图。项目用了两个大招加速：

### 🚀 大招一：Embree（Intel 出品的高性能光线追踪库）

```
没有 Embree： 你手写代码判断"这条光线撞到哪个三角形了"
              → 遍历几百万个三角形 → 慢死了

用 Embree：   Intel 帮你封装好了
              → 内部用 BVH 树 + SIMD 指令集
              → 比你手写的快 10~100 倍
```

**使用流程（极简描述）：**

```
① 创建 Embree 设备
② 把三角形数据传给 Embree
③ 让 Embree 建加速结构
④ 调用 rtcIntersect1() 算光线求交
⑤ 拿结果（撞到哪个三角形、撞在哪、距离多远）
```

> Embree 就像一个**刀具大厂**，你想切菜不用自己打铁——直接买他们的刀就行。

### 🚀 大招二：OpenMP 并行计算（多核 CPU 加速）

光子发射是**天然可并行的**——100 万个光子互不影响，可以分给 8 个 CPU 核心同时算。

```
没有并行：  100万个光子一个一个追踪 → 耗时 60 分钟
用 OpenMP：  8 核平分，每核只追踪 12.5 万个 → 耗时 8 分钟
```

关键技巧：
- 每个线程要有**独立的随机数生成器**（否则随机数不随机）
- 多个线程同时往光子数组里写数据时，需要**加锁保护**（`#pragma omp critical`）

## 九、光子映射 vs 路径追踪

你可能会问：路径追踪（Path Tracing）不是更流行吗？为什么还要学光子映射？

| 对比项 | 光子映射 | 路径追踪 |
|--------|---------|---------|
| 🎯 **方法论** | 两步走（先扔光子再查） | 一步到位（从相机发光线，光线自己找光源） |
| 💡 **焦散效果** | ✅ **天生擅长**（焦散光子图） | 🟡 需要大量采样才能清晰 |
| 🏗️ **间接光照** | ✅ **最终聚集**质量高 | ✅ 但噪声较多 |
| 🔄 **实现复杂度** | ⚠️ 高（两张光子图+KD树+最终聚集） | ✅ 相对简单 |
| ⏱️ **速度** | ⚠️ 建光子图需要时间 | ⚠️ 都需要大量采样 |
| 📊 **存储** | ⚠️ 需要存储光子图 | ✅ 不需要额外存储 |

**一句话选哪个：**

> 场景里有**玻璃杯、水面、钻石**这类会产生漂亮光斑的物体 → **光子映射**更合适
>
> 场景里大多是**漫反射材质** → **路径追踪**更简单

不过现代的渲染器往往**两者都有**，甚至混合使用（比如漫反射用路径追踪，焦散用光子映射）。

## 十一、总结与展望

### 一句话总结

> **光子映射 = 先往场景里扔光子记录"光路历史" → 再让相机光线查询"历史记录"重建间接光照。**

### 它的核心贡献

| 贡献 | 为什么重要 |
|------|-----------|
| ✅ 分离光源→场景和相机→场景 | 让间接光照可以**复用**（光子图建一次，多帧使用） |
| ✅ 全局+焦散双光子图 | 焦散效果**分离处理**，质量远超纯路径追踪 |
| ✅ KD-Tree 加速 | 百万级别光子的近邻查询做到**毫秒级** |
| ✅ 最终聚集 | 消除光子映射的"块状噪声"，达到**照片级质量** |

### 与 DSEngine 的关系

DSEngine 目前使用的是 **Forward 光栅化渲染**（`ForwardScenePass`），通过 Shadow Mapping + PBR 实现实时渲染。如果未来需要**离线烘焙**（Baking）功能，比如：

- 烘焙**光照贴图**（Lightmap）
- 预计算**环境光遮蔽**（AO）
- 生成**高质量焦散贴图**

光子映射是非常合适的技术选型——但 DSEngine **暂未集成 Embree 或 OptiX 加速库**，如果要用光子映射做离线烘焙，需要先引入外部光线求交库（如 Intel Embree 或直接使用 CPU 暴力求交），属于中等复杂度的扩展。
