## 一、渲染管线是什么？—— 3D 变 2D 的"魔法工厂流水线"

先来一张"灵魂理解图"：

> **渲染管线 = GPU 把 3D 场景变成 2D 图像的整条流水线**

想象你开了一家玩具工厂：

1. **设计图纸来了**（游戏程序准备数据）
2. **把图纸上的零件搬到生产线上**（把顶点、纹理数据送到显存）
3. **在传送带上把 3D 立体零件压扁成 2D 平面图纸**（几何阶段）
4. **决定哪些格子（像素）要涂上颜色**（光栅化阶段）
5. **给每个格子涂上正确的颜色**（像素处理阶段）

最终，你屏幕上看到的每一个像素，都是这条流水线的产物。

**流水线的四个大步骤：**

| 阶段 | 通俗理解 | 谁干的 |
|------|---------|--------|
| ① 应用阶段 | 程序员写好代码，准备好场景数据 | CPU |
| ② 几何阶段 | 把 3D 物体压扁到 2D 屏幕上 | GPU |
| ③ 光栅化阶段 | 判断屏幕每个像素应该是什么颜色 | GPU |
| ④ 像素处理阶段 | 深度测试、混合、最终输出画面 | GPU |

## 二、应用阶段——程序员在 CPU 上做的准备工作

这个阶段主要是在 **CPU** 上发生的，程序员写的游戏逻辑在这里起作用。

### 2.1 尽量减少 GPU 的"工作量"

GPU 很贵（卡贵、电费贵），所以我们要帮它省点事。怎么做呢？

**三种"剔除"技巧：**

1. **视锥剔除**——摄像机看不见的物体就不渲染
   - 你可以想象自己拿着手电筒，照不到的地方就干脆不管了

2. **遮挡剔除**——被墙挡住的物体也不渲染
   - 就像你站在一堵墙前面，墙后面的东西你看不到，那就不用画

3. **背面剔除**——物体背对你的三角形也不渲染
   - 一个魔方从外面看，你永远看不到魔方内部的面

**关于背面剔除的小知识：**
三角形的顶点顺序如果是**顺时针**，说明它在背面，会被 GPU 忽略掉。这是用"叉乘"算出来的方向（右手定则，后面会讲）。

### 2.2 LOD 优化——近处用高清，远处用标清

LOD 的全称是 **Levels of Detail**（细节级别）。

- 离你 1 米远的树：用几万个三角形，树叶脉络清晰可见
- 离你 100 米远的树：用几十个三角形，看着像一坨绿色就行
- 离你 500 米远的树：甚至可以用一张纸片（面片）

这样 GPU 就不用每次都渲染最精细的模型，省下大量性能。

### 2.3 旋转的三种方式——欧拉角、矩阵、四元数

游戏里的物体经常要旋转，比如角色转个身、风扇转起来。有三种主流方式：

| 方法 | 通俗理解 | 优点 | 缺点 |
|------|---------|------|------|
| **欧拉角** | 说"绕X轴转30°，再绕Y轴转60°" | 简单直观，人类看得懂 | 会"万向锁"（两个轴卡死），不同顺序结果不同 |
| **矩阵** | 用 4×4 的数字表格表示变换 | 啥都能做（平移旋转缩放），计算灵活 | 占内存（16个float），不直观 |
| **四元数** | 用 (x,y,z,w) 四个数表示绕哪个轴转多少 | 没有万向锁，能做平滑插值，性能好 | 很不直观，正常人看不懂 |

**实际开发中的用法：**
- **人看的时候** → 用欧拉角（给策划调参数用）
- **底层计算** → 用四元数（给程序用）
- **传给渲染管线** → 转成矩阵（给 GPU 用）

## 三、几何阶段——把 3D 数据变成屏幕上的坐标

几何阶段就是把 3D 世界里的物体"拍扁"到 2D 屏幕上。

### 3.1 顶点着色器——最重要的一步

顶点着色器干两件事：

1. **变换顶点位置**（用 MVP 矩阵）
2. **准备顶点颜色数据**（给后续插值用）

#### 啥是 MVP 矩阵？三个矩阵打天下！

**M（Model，模型矩阵）：把物体从"自己的坐标系"放到"世界坐标系"**

- 在建模软件里，茶壶在原点 (0,0,0)
- 在游戏世界里，茶壶在桌子上的 (5, 2, 3)
- M 矩阵干的就是"搬家"这件事

**V（View，视图矩阵）：把世界坐标变成"以摄像机为原点"的坐标**

- 想象你拿着自拍杆拍照，V 矩阵就是在算你（摄像机）在世界里的位置
- 最终效果：不管摄像机怎么动，都相当于世界在反着动

**P（Projection，投影矩阵）：把 3D 空间压扁成一个立方体"

- 透视投影：**近大远小**（真实相机效果）
- 正交投影：**没有近大远小**（工程图纸效果）

> ❓ **问：MVP 操作后还有近大远小效果吗？**
>
> 答：没有了。投影矩阵已经把视锥体压成了一个立方体，在这个过程中"近大远小"已经算好了。后面处理的是这个立方体里的数据。

### 3.2 NDC 归一化——统一到一个标准立方体

NDC（Normalized Device Coordinates，标准化设备坐标）就是把所有坐标都归一化到 `[-1, 1] × [-1, 1] × [-1, 1]` 这个标准立方体中。

这个过程包含三步：

1. **裁剪**——把视锥体外的三角形扔掉
2. **透视除法**——除以 w，得到真正的 3D 坐标
3. **屏幕映射**——从 NDC 坐标变成屏幕像素坐标

> ❓ **问：之前不是已经"视锥剔除"了吗？为啥还要再裁剪一次？**
>
> 答：视锥剔除是针对**整个物体**（比如整棵树）。NDC 裁剪是针对**单个三角形**。一个物体可能一半在屏幕内一半在外面，物体级别的剔除不够细，需要到三角形级别的裁剪。

> ❓ **问：为啥要用"齐次坐标"（4 个数的坐标）这么麻烦的东西？**
>
> 答：因为用 4 维坐标可以把**平移**也统一成矩阵乘法（3 维搞不定平移）。点的 w = 1，方向的 w = 0（方向不会被平移影响）。

## 四、光栅化阶段——决定哪些像素要被画出来

"光栅"就是"网格"的意思，"光栅化"就是把三角形变成屏幕上一个个像素格子的过程。

想象你在方格纸上画一个三角形：

- 三角形是**连续**的（平滑的线条）
- 屏幕像素是**离散**的（一个个小方格）
- 怎么决定哪个方格涂色？这就是光栅化要干的事

### 4.1 三角形设置 + 三角形遍历

**三角形设置：** GPU 先算出三角形的三条边的方程，以及颜色、纹理坐标等属性的插值公式。

**三角形遍历：** 判断每个像素的中心点**是否在三角形内部**，是的话就保留，不是就丢弃。

### 4.2 怎么判断一个点在三角形内部？

有三种方法，GPU 最常用的是第一种：

**① 重心坐标法（最常用）**

把点 P 表示为三个顶点的"加权和"：`P = uA + vB + wC`

其中 `u + v + w = 1`，如果 u、v、w 都在 [0, 1] 范围内，说明 P 在三角形内部。

为啥 GPU 偏爱这种方法？因为不仅能判断是否在内部，还能顺便算出**插值权重**，用来算这个点的颜色、纹理坐标等。

**② 叉积法（同侧法）**

依次判断 P 是否在每条边的同一侧。如果在三条边的同一侧，说明在内部。

**③ 面积法**

计算三个小三角形的面积之和是否等于原三角形面积，如果相等则在内部。

### 4.3 片段（Fragment）是什么？

被三角形覆盖的每个像素会生成一个"片段"（也叫片元）。一个片段包含：

- 屏幕坐标 (x, y)
- 深度值 z（离摄像机多远）
- UV 坐标（纹理位置）
- 法线方向
- 颜色值

> ⚠️ 一个像素可能对应**多个片段**（多个三角形叠在一起），最终在深度测试时决定要哪个。

### 4.4 快速复习：点乘和叉乘

**点乘（Dot Product）：结果是一个数值**

公式：`A · B = |A| × |B| × cos θ`

用途：算投影、算两个向量的夹角

**叉乘（Cross Product）：结果是一个向量**

公式：`A × B` 的结果垂直于 A 和 B 所在的平面

用途：算法线方向、判断点在三角形哪一侧

> 小技巧：右手定则——右手四指从 A 弯向 B，大拇指方向就是叉乘结果的方向。

## 五、像素处理阶段——最终给每个像素上色

这是流水线的最后一步，**决定每个像素最终显示什么颜色**。

### 5.1 片段着色器——给片段"化妆"

拿到光栅化后的片段数据，片段着色器会：

1. **纹理采样**——根据 UV 坐标从贴图上读取颜色
2. **光照计算**——算算这个点受光后该是什么颜色
3. **材质计算**——表面是粗糙还是光滑？金属还是木头？
4. **其他效果**——阴影、环境光遮蔽等

> ⚠️ **片段着色 ≠ 像素着色**。片段着色只是像素处理阶段的一步，后面还要经过深度测试、模板测试、混合等步骤，才能最终得到像素颜色。

### 5.2 逐片段操作——谁留谁走？

**① 深度测试（Z-Buffer）——解决遮挡问题 —— 核心详解**

深度测试是渲染管线中**最基础也最重要的环节之一**。没有它，远处的物体会画在近处物体之上，画面就会一团糟。

#### 深度测试的核心思想

> **每个像素只显示离摄像机最近的物体的颜色。**

GPU 为此维护了一张"深度缓冲区"（Z-Buffer / Depth Buffer），它和屏幕上的颜色缓冲区一一对应——**每个像素除了存颜色，还存一个深度值**。

#### 深度值表示什么？

| 概念 | 含义 |
|------|------|
| **深度值 (Z-value)** | 片段到摄像机的距离，归一化到 [0, 1] 范围 |
| **0.0** | 离摄像机**最近**（就在镜头前面） |
| **1.0** | 离摄像机**最远**（在远裁剪面） |
| **深度越小 → 越靠前** | 会挡住深度更大的片段 |

#### 工作流程——"谁更近谁赢"

想象你透过窗户看外面的一排树：

```
摄像机 ←→ 第1棵树 (z=0.2) ←→ 第2棵树 (z=0.5) ←→ 第3棵树 (z=0.8)
```

GPU 处理这个像素的过程：

```
┌──────────────────────────────────────────────────────────┐
│  步骤 1: 画第1棵树                                        │
│    → 深度 0.2 < Z-Buffer 默认值 1.0                       │
│    → ✅ 通过！写入颜色，Z-Buffer 更新为 0.2                │
├──────────────────────────────────────────────────────────┤
│  步骤 2: 画第2棵树                                        │
│    → 深度 0.5 > Z-Buffer 中的 0.2                         │
│    → ❌ 被挡住！丢弃这个片段                               │
├──────────────────────────────────────────────────────────┤
│  步骤 3: 画第3棵树                                        │
│    → 深度 0.8 > Z-Buffer 中的 0.2                         │
│    → ❌ 被挡住！丢弃这个片段                               │
├──────────────────────────────────────────────────────────┤
│  最终屏幕: 显示第1棵树的颜色                                │
└──────────────────────────────────────────────────────────┘
```

#### 用代码表示

```glsl
// 伪代码：深度测试的逻辑
float currentDepth = fragment.depth;          // 当前片段的深度
float bufferDepth = zBuffer[pixel.x][pixel.y]; // Z-Buffer 里存的深度

if (currentDepth < bufferDepth) {
    // ✅ 当前片段更近，覆盖颜色和深度
    framebuffer[pixel.x][pixel.y] = fragment.color;
    zBuffer[pixel.x][pixel.y] = currentDepth;
} else {
    // ❌ 当前片段被挡住了，直接丢弃
    discard;
}
```

#### 为什么叫 Z-Buffer？

在 3D 数学中，三个坐标轴是 X（左右）、Y（上下）、**Z（前后——指向屏幕里）**。Z 值就是"深度"，所以叫 **Z-Buffer**（Z 缓冲区），也叫**深度缓冲**。

#### 为什么 Z-Buffer 默认值是 1.0 而不是 0？

因为深度值越小表示越近，默认初始化为最大值 1.0（最远），这样第一个片段无论多深都能通过测试。相当于"目前最近的是无穷远，你来你就赢"。

#### 生活中最像什么？

> 想象你面前站了一排人，你想看清**最前面**那个人是谁。你手里拿着一张名单，每看到一个人，就记下"这个人比之前那个更靠前吗？是的话就替换"。最后名单上留下的就是最前面那个人。

Z-Buffer 就是 GPU 手里的那张"谁更近"的名单。

#### 深度测试的精度问题

你可能觉得奇怪：深度值范围只有 [0, 1]，但场景距离可能从 1 米到 1000 米，精度够吗？

答案是：**近处精度高，远处精度低**。这是 Z-Buffer 的固有特性。

原因是深度值在 NDC 中的分布是**非线性的**：

```
近处 (z=1m)   → 深度值变化明显 → 精度高，区分度好
远处 (z=100m) → 多个深度值挤在一起 → 精度低，容易出现"深度冲突"
```

**Z-Fighting（深度冲突）**：当两个物体的表面几乎在同一位置时，深度值因精度问题难以区分，导致像素在两个颜色之间疯狂**闪烁**。远处墙面或地面经常出现这个问题。

解决方法：
- 使用**反向 Z-Buffer**（把远近颠倒，远处精度变高）
- 使用更高精度的深度缓冲区（24-bit → 32-bit）
- 尽可能让近裁剪面远离摄像机，缩小深度范围

**② 模板测试——自定义蒙版**

就像 Photoshop 里的蒙版，你可以定义屏幕的哪些区域能画、哪些不能画。常用于镜子效果、特效遮罩等。

**③ 混合（Blending）——处理透明物体**

半透明物体不直接覆盖，而是和背景颜色**混合**。公式：

```
最终颜色 = 新颜色 × 透明度 + 背景颜色 × (1 - 透明度)
```

比如一层有色玻璃，既要看到玻璃的颜色，又要透过玻璃看到后面的东西。

**④ 写入帧缓冲区**

最终结果写入 Framebuffer，GPU 再把 Framebuffer 的内容发送给显示器，你就看到画面了！

### 5.3 各个测试的顺序

```
裁剪测试 → Alpha测试 → 模板测试 → 深度测试 → 混合
```

**为啥不能先选好片段再着色？**

因为有这几种情况：
1. 一些特殊效果需要在着色器里修改深度值，必须先着色
2. 一些片段可能在着色器里被丢弃（比如透明测试），提前选会选错
3. 透明物体需要混合多个片段，没法提前确定选哪个

**Early-Z（提前深度测试）：** 一种优化技巧，在着色前先做深度测试，把被挡住的片段提前丢掉。但只有着色器没有修改深度、没有丢弃片段时才能用。

### 5.4 透明混合的注意事项

半透明物体需要**从后往前画**，否则混合效果会出错。而且通常只做深度测试（判断谁在前面），但不写入深度值（因为透明物体不遮挡后面的东西）。

## 六、走样与抗锯齿——为啥会有锯齿？怎么消除？

### 6.1 什么是"走样"？

走样（Aliasing）就是**采样速度跟不上信号变化速度**导致的失真。

**常见的四种走样：**

1. **几何走样**——斜线变成锯齿（最常见）
   - 就像用低分辨率画一条斜线，会看到"楼梯状"的锯齿

2. **纹理走样**——远处的纹理出现摩尔纹
   - 就像穿条纹衬衫上电视，有时候会看到奇怪的波纹

3. **Shader 走样**——高光闪烁
   - 高光范围太小，采样时有时无

4. **时间走样**——车轮看起来在反转
   - 电影里马车的轮子有时候看起来在倒转，就是采样频率（帧率）跟不上轮子转速

### 6.2 抗锯齿技术——把锯齿磨平

**① SSAA（超级采样抗锯齿）**

思路：先在 2 倍的画布上渲染，再缩回到原来的分辨率。

效果最好，但性能最差（要渲染 4 倍的像素，直接掉帧）。

**② MSAA（多重采样抗锯齿）**

思路：一个像素内多取几个点判断覆盖率，最后按覆盖率混合颜色。

比 SSAA 快很多，是传统方案中性价比不错的。

**③ FXAA（快速近似抗锯齿）**

思路：在最终画面上直接处理，找到锯齿边缘然后模糊掉。

速度很快，但会让画面变模糊。

**④ TAA（时间抗锯齿）**

思路：利用过去几帧的信息来帮助当前帧抗锯齿。

性能开销小，是现代游戏的主流选择。

**⑤ DLSS（深度学习超采样）**

思路：用 AI 深度学习，从低分辨率渲染结果"猜"出高分辨率画面。

NVIDIA 的看家本领，画质好性能也高。

### 6.3 纹理走样的解决方法：Mipmap

还记得前面说的"摩尔纹"吗？一个屏幕像素对应了太多纹理像素，直接采样就会出问题。

**Mipmap 纹理金字塔** 的解决方案：

- 提前把纹理缩小成不同分辨率：原图 → 1/2 → 1/4 → 1/8 ...
- 远处物体用低分辨率纹理，近处用高分辨率
- 额外内存只增加约 **1/3**，性价比很高

**各向异性过滤：** Mipmap 只能处理正方形区域，但如果纹理是斜着的（一个像素覆盖了长方形的纹理区域），就需要各向异性过滤。它不占用额外内存，只是增加采样计算。

## 七、光照模型与阴影——物体为什么"亮"和"暗"？

### 7.1 三种经典光照模型

**① Lambert 漫反射模型（最简单）**

想象一张粗糙的白纸，不管从哪个角度看，亮度都一样。

- 只和光线方向与表面法线的夹角有关
- 光线越斜越暗，背光就全黑
- 没有高光

**② Phong 反射模型（经典三重奏）**

把光照分成三部分加起来：

1. **环境光**——即使没有直接光照，场景也不是全黑的（模拟光线在环境里反弹）
2. **漫反射**——和 Lambert 一样，各个方向看亮度一致
3. **高光（镜面反射）**——光滑表面的反光，随着视角变化

> Phong 的缺陷：当视线和反射光夹角>90°时，高光会突然消失，看起来不自然。

**③ Blinn-Phong 反射模型（改进版）**

改进了 Phong 模型中高光的计算方式。

不直接算反射光线了，而是引入**半程向量 H**（光线方向和视线方向的中间方向）。H 和法线 N 越接近，高光越强。

- 算得更快（省去了反射向量的计算）
- 数值更稳定（不会出现角度>90°的问题）

---

### 7.2 阴影——怎么判断一个点被光照到了？

光照模型假设"光一定能照到物体表面"，但现实中有遮挡啊！于是需要专门的阴影技术。

**① 光照贴图（Lightmap）——静态场景的神器**

- 提前"烘焙"好场景的光照信息，存到一张贴图里
- 运行时直接采样，不用每帧计算
- 缺点：只能用于**静态物体**（桌子椅子不能动）

**② 阴影贴图（Shadowmap）——动态阴影的经典方案**

- 从光源位置"看"一次场景，记录看到的最远深度
- 渲染时比较当前像素的深度和记录的深度
- 如果当前深度 > 记录的深度 → 在阴影里

> 小贴士：游戏里的"全局光照"其实没真正把所有光线的路径算一遍（太费性能了），而是用各种近似方法模拟的。所以即使开了全局光照，还是需要 Shadowmap。

**Shadowmap 的两个常见问题：**

1. **阴影条纹（自阴影）**——浮点精度不够，物体会给自己投出条纹阴影
   - 解决方法：加一个 bias（偏移量）再比较
   - 但要小心偏移太大了阴影会"飘"起来

2. **阴影锯齿**——Shadowmap 本质也是一张纹理，也会有锯齿
   - 解决方法：PCF（百分比渐进过滤），类似 MSAA，多次采样平均

## 八、全局光照——光是怎样满屋子乱跑的？

光照模型说的是"一个表面怎么反射光"，全局光照说的是"光在整个场景里怎么传播"。

**简单理解：**

- 光照模型 → 告诉你颜料盒里每种颜料是啥颜色
- 全局光照 → 告诉你应该蘸哪个颜料、蘸多少

### 8.1 四个基础概念（记住这几个名词就够了）

1. **辐射通量**——光源每秒钟发出的总能量（相当于灯泡的瓦数）
2. **辐射强度**——某个方向上有多亮
3. **辐照度**——某个表面接收到了多少光
4. **辐射率**——某个表面朝某个方向发出了多少光

### 8.2 BRDF——材质的"身份证"

BRDF（双向反射分布函数）描述的是：**光从一个方向照到表面后，会以什么方式反射到其他方向。**

你可以把它理解为材质的"反射性格"：
- 粗糙木头：光往各个方向均匀反射（漫反射强）
- 光滑金属：光集中在特定方向反射（高光强）

### 8.3 渲染方程——终极光照公式

渲染方程描述的是"从某个方向看一个表面点，能看到多少光"。

公式看起来挺吓人的，但核心思想很简单：**一个点反射到眼睛的光 = 这个点自身发的光 + 所有方向照过来的光 × BRDF × 夹角影响**

### 8.4 光线追踪 vs 路径追踪

**光线追踪：**
- 从摄像机出发，每个像素发出一条或多条光线
- 光线碰到物体，根据 BRDF 计算反射方向，继续追踪
- 直到光线离开场景、打到光源、或者达到最大递归次数

**路径追踪（光线追踪的升级版）：**
- 光线追踪有个问题：漫反射面不会继续发射光线，导致漫反射效果算不准
- 路径追踪用**蒙特卡洛方法**（随机采样）来解决
- 碰到漫反射面时，在半球上**随机选一个方向**继续追踪
- 多条随机路径的结果平均起来，就是最终亮度

> 光线追踪效果好，但是**非常非常慢**，一般用于离线渲染（做电影、做效果图）。实时游戏用的都是各种"取巧"的近似方案。

## 九、空间加速结构——怎么让电脑"少做无用功"？

在 3D 场景中，判断两个物体是否碰撞、判断光线和哪个物体相交，如果一个个去算，性能会爆炸。

**核心思路：先粗筛再细查**

### 9.1 空间划分——把空间切块

**① 均匀网格**
- 把世界切成一样大的小格子
- 只检测光线穿过的格子里的物体
- 简单、常用，但格子大小不好控制

**② 八叉树**
- 递归地把空间切成 8 块
- 物体多的区域切得更细
- 2D 版本叫"四叉树"

**③ BSP 树**
- 用平面把空间切成两半，每半再接着切
- 切得高效但实现复杂，物体移动后要重建

**④ KD 树（BSP 的简化版）**
- 只沿坐标轴方向切（先沿 X 切，再沿 Y 切，再沿 Z 切）
- 实现比 BSP 简单，动态更新依然困难

### 9.2 物体划分——把物体包起来

**BVH（层次包围盒）——现代游戏/引擎的主流方案**

不切空间，而是**包物体**：

1. 把场景里所有物体用一个大的包围盒包起来
2. 按位置分成两组，分别用两个小包围盒包起来
3. 递归下去，直到每个盒子里只有少量物体

检测时：
- 先判断是否和大包围盒相交
- 不是 → 直接结束
- 是 → 继续判断子包围盒

BVH 的优点是**构建快、支持动态场景**，非常适合游戏这种物体经常移动的场合。

### 9.3 包围盒的三种类型

**① AABB（轴对齐包围盒）——最常用**
- 边永远平行于坐标轴
- 哪怕物体旋转了，包围盒也不转（只变大）
- 只需要两个点就能表示，存储小，计算快

**② OBB（有向包围盒）**
- 可以跟着物体一起旋转
- 更贴合物体，但碰撞检测复杂

**③ 包围球**
- 一个球把物体包起来
- 碰撞检测最简单（算距离就行）
- 但空隙大，不太精确

## 十、延迟渲染与 PBR——现代游戏的"黑科技"

### 10.1 前向渲染 vs 延迟渲染

**前向渲染（传统方案）：**
```
先着色 → 再深度测试
```
- 每个物体在每个光源下都要算一遍颜色
- 复杂度 = 物体数量 × 光源数量
- 简单但性能差，光源一多就扛不住

**延迟渲染（现代方案）：**
```
先深度测试 → 再统一着色
```
- 第一步：把场景所有物体的位置、法线、材质等信息存到 G-Buffer
- 第二步：统一计算光照，只看最终可见的像素
- 复杂度 = 物体数量 + 光源数量
- 性能好，但占显存多，不支持透明混合

### 10.2 PBR（基于物理的渲染）

PBR 不是某个具体技术，而是一整套**基于物理法则的渲染框架**。

**传统渲染 vs PBR：**

- 传统：用经验公式算颜色（像 Phong 模型），看着差不多就行
- PBR：模拟真实物理过程，追求"能量守恒"，看起来更真实

**PBR 三要素：**

1. **基于物理的材质**——用粗糙度、金属度等参数描述物体
2. **基于物理的光照**——光源携带真实物理能量
3. **基于物理的相机**——曝光控制、光晕效果

**Cook-Torrance BRDF（PBR 的核心）：**
- 法线分布函数：决定高光的形状和强度（粗糙还是光滑？）
- 几何函数：考虑微表面的遮挡，防止能量溢出
- 菲涅尔方程：多少光被反射，多少被吸收（决定边缘亮度）

### 10.3 PBR 和 NPR 的区别

| | PBR（基于物理） | NPR（非真实感） |
|------|---------|--------|
| 目标 | 逼真，模拟真实世界 | 风格化，像手绘/漫画 |
| 代表游戏 | 《荒野大镖客》《巫师3》 | 《原神》《女神异闻录》 |
| 核心技术 | 能量守恒、物理材质 | 描边、卡通着色 |
| 感觉 | "哇，好真实！" | "哇，好好看！" |

很多游戏其实是**两者结合**的——用 PBR 做基础渲染，再叠加上 NPR 的风格化效果。
