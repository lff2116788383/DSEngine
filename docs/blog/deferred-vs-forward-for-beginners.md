# 延迟渲染 vs 前向渲染——游戏里的"先画再打光"和"边画边打光"

> **灵魂拷问：一个场景里有 100 盏灯，GPU 要算多少次光照？**
>
> 答案取决于你用的是**前向渲染（Forward Rendering）**还是**延迟渲染（Deferred Rendering）**。这两个名字听起来高大上，其实思路特别简单——一个像"边炒边放调料"，一个像"全部炒完再统一浇汁"。

## 一、前向渲染：每个物体自己算光照——简单直接

### 什么是前向渲染？

前向渲染是**最传统、最直观**的渲染方式。它的思路很简单：

> **你画每一个物体的时候，当场就把光照算好。**

### 用生活场景理解

🍳 **做饭类比：**

前向渲染 = **边炒菜边放调料**

- 你炒一盘青菜，放盐、放酱油、放蒜末——**每个菜单独调味**
- 如果做 10 个菜，每个菜都要单独放调料
- 调料放得好不好，取决于你这个厨师的功力

渲染里就是这样的：

```
物体 A → 算光照 → 输出到屏幕
物体 B → 算光照 → 输出到屏幕
物体 C → 算光照 → 输出到屏幕
...

每个物体都完整地算一遍光照（包括所有光源的影响）
```

### 前向渲染的执行过程

```
      场景数据（模型、材质、灯光）
              ↓
  ┌─────────────────────────────┐
  │  物体 1                      │
  │    ├─ Vertex Shader          │
  │    ├─ 光栅化                 │
  │    └─ Fragment Shader        │
  │       ├─ 采样纹理           │
  │       ├─ 计算光照（光源1~N）│  ← 每个物体都要算所有灯
  │       └─ 输出颜色           │
  ├─────────────────────────────┤
  │  物体 2                      │
  │    └─ 同上流程              │
  ├─────────────────────────────┤
  │  ... 以此类推                │
  └─────────────────────────────┘
              ↓
          最终画面
```

### 前向渲染的优点

| 优点 | 说明 |
|:----|:------|
| ✅ **实现简单** | 代码逻辑直观，容易理解 |
| ✅ **支持透明物体** | 透明度可以正确处理 |
| ✅ **支持抗锯齿（MSAA）** | 硬件 MSAA 在前向渲染上效果最好 |
| ✅ **显存占用低** | 不需要额外的中间缓存 |
| ✅ **移动端友好** | 低功耗设备上的首选 |

## 二、问题来了：100 盏灯，每个物体都要算 100 次——性能炸了

### 数学题

前向渲染有一个致命问题——**计算量随着光源数量线性增长**。

假设场景里有：
- 500 个物体
- 100 盏灯

前向渲染要做的事情：

```
每个 Fragment Shader 里：
  光照计算次数 = 物体数 × 光源数

在 1920×1080 分辨率下：
  总像素 ≈ 200 万
  如果每个像素都受所有灯影响：
  光照计算 = 200万 × 100 灯 = 2亿 次

每秒 60 帧：
  每秒光照计算 = 2亿 × 60 = 120亿 次
```

> **显卡：我谢谢你啊。**

### 为什么旧游戏灯不多？

你回忆一下 2010 年以前的游戏：

- 一个场景里可能只有 2~4 盏灯
- 大多数是"方向光"（太阳），算一次就行
- 因为都是前向渲染，灯多了扛不住

这就是前向渲染的最大瓶颈：**每多一盏灯，整个场景的计算量就要翻倍。** 100 盏灯？显卡会直接躺平。

### 前向渲染的优化方案

为了让前向渲染多撑几盏灯，人们想了一些办法：

**① 只算最近的几盏灯**
- 每个物体只被最近的 4~8 盏灯照亮
- 远处的灯对你影响太小，忽略不计
- 缺点：灯光切换时画面会"跳变"

**② 分 Pass 渲染**
- 物体先画一次（主光），再画第二次（补光），再画第三次……
- 每个 Pass 算一盏灯，最后颜色加起来
- 缺点：物体要画 N 遍，Draw Call 翻 N 倍

这些方案只能缓解，不能根治。直到**延迟渲染**出现，才彻底解决了这个问题。

## 三、延迟渲染：先把物体存进 GBuffer，再统一算光照

### 核心思路

> **延迟渲染的精髓就四个字：先存后算。**

把渲染分成两大步：

```
第一步：只存不算法
  把每个像素的"基本信息"存到几张临时纹理里
  不计算光照！
  
第二步：统一算光照
  拿着存好的信息，一次性把所有像素的光照算完
```

### 用生活场景理解

🐟 **做饭类比 2.0：**

前向渲染 = 边炒菜边放调料 → **每人一份，单独调味**

延迟渲染 = **麻辣烫统一浇汁**
- 第一步：把所有食材（肉片、豆腐、蔬菜）都烫熟，摆好
- 第二步：调一大锅麻辣汤汁，统一浇在所有食材上
- 不管你有 10 种食材还是 100 种食材，汤汁只调一次

### 延迟渲染的执行过程

```
Pass 1：几何 Pass（G-Pass）
  只存信息，不算光照
  
  ┌─────────────────────────────┐
  │  GBuffer（几何缓冲区）       │
  │  ├─ 颜色纹理（Albedo）      │
  │  ├─ 法线纹理（Normal）      │
  │  ├─ 位置/深度（Position/Depth）│
  │  ├─ 材质参数（PBR 参数）    │
  │  └─ ...                      │
  └─────────────────────────────┘
              ↓
Pass 2：光照 Pass（Lighting Pass）
  统一算光照，一次搞定

  ┌─────────────────────────────┐
  │  从 GBuffer 读取信息         │
  │  ├─ 位置 → 判断离灯多远      │
  │  ├─ 法线 → 判断光照角度      │
  │  ├─ 材质 → 判断怎么反光      │
  │  └─ 计算最终颜色             │
  └─────────────────────────────┘
              ↓
          最终画面
```

## 四、GBuffer 是什么？——几个"图层"各存各的信息

### GBuffer 就是"信息仓库"

GBuffer 的全称是 **Geometry Buffer**（几何缓冲区）。它不是一张图，而是一组图——每张图存一种信息。

想象你在用 Photoshop 做设计：

```
图层 1（Albedo）：       ┌──────────────────┐
  物体的"素颜照"         │  红色的墙         │
  不包含光照信息          │  蓝色的天空       │
                        │  绿色的草地       │
                        └──────────────────┘

图层 2（Normal）：        ┌──────────────────┐
  每个像素的"朝向"        │  朝上的法线→绿色  │
  红色=X方向，绿色=Y     │  朝右的法线→红色  │
  蓝色=Z方向             │  朝前的法线→蓝色  │
                        └──────────────────┘

图层 3（Depth）：         ┌──────────────────┐
  每个像素离相机多远      │  近处→白色        │
  白色=近，黑色=远       │  远处→黑色        │
                        │  渐变过渡          │
                        └──────────────────┘

图层 4（Material）：      ┌──────────────────┐
  金属度+粗糙度          │  金属→亮          │
                        │  粗糙→暗          │
                        └──────────────────┘
```

### GBuffer 为什么是"革命性"的？

前向渲染里，一个像素的光照计算发生在 Fragment Shader 里——这意味着**如果 100 个物体叠在同一个像素上（深度测试后只留最前面的），前面 99 个物体的光照计算就白做了！**

延迟渲染解决了这个问题：

> **先只管"谁在最前面"（深度测试），只存最前面那个像素的信息到 GBuffer。然后只用 GBuffer 里的数据算光照——前面的计算一个都没浪费。**

## 五、延迟渲染的优缺点：性能好但吃显存

### 优点

| 优点 | 详细说明 |
|:----|:--------|
| ✅ **光源数量不限制** | 100 盏灯和 1000 盏灯，性能开销差不多 |
| ✅ **每像素只算一次光照** | 不会因为物体遮挡而浪费计算 |
| ✅ **适合复杂场景** | 大量动态光源的场景，延迟渲染完胜 |
| ✅ **后处理友好** | GBuffer 里的法线/深度信息可以直接给后处理用 |

### 缺点

| 缺点 | 详细说明 |
|:----|:--------|
| ❌ **显存占用大** | GBuffer 要多张高精度纹理，4K 下一张就要几十 MB |
| ❌ **不支持 MSAA** | 硬件 MSAA 对 GBuffer 无效（但可以用 TAA 替代） |
| ❌ **透明物体很难处理** | 透明度需要排序，延迟渲染做不了 |
| ❌ **带宽要求高** | 读写 GBuffer 需要大量显存带宽 |
| ❌ **移动端不友好** | 功耗和带宽都很紧张 |

## 六、对比表格：前向 vs 延迟

| 对比维度 | 前向渲染 | 延迟渲染 |
|:--------|:--------|:--------|
| **核心思路** | 边画边算光 | 先存再算光 |
| **光源数量** | ❌ 对光源数量敏感（4~8 盏就吃力） | ✅ 支持大量光源（100+ 盏无压力） |
| **性能瓶颈** | 光源数 × 物体数 | 分辨率（像素越多越慢） |
| **MSAA 支持** | ✅ 完美支持 | ❌ 不支持 |
| **透明物体** | ✅ 可以正确处理 | ❌ 需要特殊处理 |
| **显存占用** | ✅ 低 | ❌ 高（多张 GBuffer） |
| **移动端** | ✅ 首选方案 | ❌ 功耗高，带宽不够 |
| **实现复杂度** | ✅ 简单 | ❌ 复杂 |
| **适用场景** | 光源少、移动端、VR | 光源多、AAA 大作、PC/主机 |

### 图形化直观对比

```
前向渲染的"浪费"：
  ┌──────────┬──────────┬──────────┐
  │ 物体1    │ 物体2    │ 物体3    │
  │ 算10盏灯 │ 算10盏灯 │ 算10盏灯 │ ←每个物体都算10盏灯
  │    ↓     │    ↓     │    ↓     │
  │ 被遮挡！  │ 被遮挡！  │ 显示     │ ←前两个白算了
  └──────────┴──────────┴──────────┘

延迟渲染的"高效"：
  ┌──────────────────────────────┐
  │ 几何 Pass：只存信息，不算光   │
  │ 物体1、2、3 → 只存最前面的    │
  ├──────────────────────────────┤
  │ 光照 Pass：一次算所有像素     │
  │ 从 GBuffer 读取 → 统一打光   │
  │ 不管前面挡了多少物体          │
  │ 只算最终显示的那个像素！      │
  └──────────────────────────────┘
```

## 七、现实中的方案：两种都用

### 没有银弹

现实中没有完美的方案。现代游戏引擎的处理方式是：

> **小孩子才做选择，成年人全都要。**

### 混合渲染（Hybrid Rendering）

大多数现代引擎（包括 DSEngine）采用的是**混合方案**：

```
                场景
                  │
           ┌──────┴──────┐
           │              │
     不透明物体         透明物体
           │              │
     延迟渲染 ╱          ╲ 前向渲染
     （主力）            （处理透明）
           │              │
           └──────┬──────┘
                  │
             合并输出
```

具体来说：

| 物体类型 | 渲染方式 | 原因 |
|:--------|:--------|:------|
| 不透明物体（墙、地面、角色身体） | 延迟渲染 | 占绝大多数，需要多光源 |
| 透明物体（玻璃、水面、粒子） | 前向渲染 | 透明度需要排序，延迟做不了 |
| 半透明特效（火焰、烟雾） | 前向渲染 | 同上 |
| UI、文字 | 前向渲染 | 简单、不需要光照 |

### 另一种方案：Clustered Forward+

还有一种叫做 **Clustered Forward+** 的技术——它是前向渲染的"增强版"：

> **把屏幕分成一个个小格子（Cluster），每个格子只计算能照到它的那几盏灯。**

比如 100 盏灯，大部分灯离这个格子很远，真正有影响的可能只有 5 盏。这样既保留了前向渲染的优点（支持透明、支持 MSAA），又能处理大量光源。

## 八、一句话总结

> **前向渲染是"边炒菜边放调料"——每个物体单独算光照，简单直接但扛不住多光源；延迟渲染是"统一浇汁"——先存好所有材质信息再统一算光照，多光源无压力但吃显存、不支持透明。现代游戏的做法是两者混用：不透明物体用延迟，透明物体用前向。**
>
> **所以下次玩游戏看到满屏霓虹灯还不卡的时候，记得感谢延迟渲染——它让你的显卡少干了 90% 的活。** 🎮
