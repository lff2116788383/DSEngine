# RTXGI-DDGI：NVIDIA 的实时全局光照"黑科技"，到底是怎么工作的？

> 一篇写给小白的实时全局光照入门指南

---

## 一、先从一个问题说起：为什么游戏里的光照总是不够"真"？

玩 3A 游戏时，你有没有注意到这个现象：

> 你走进一个房间，阳光从窗户照进来。现实里，**整个房间都会被照亮**——天花板反射一些光到墙上，墙上再反射到沙发背面，沙发背面的光又弹到角落……
>
> 但在游戏里，**窗户照不到的地方常常一团漆黑**。

**为什么？**

因为传统游戏渲染（光栅化）只计算"直接光照"——光线从光源➡️物体➡️眼睛，**只弹一次**。

而现实中，光线会**弹无数次**。这就是**全局光照（Global Illumination，GI）**要做的事。

### 🔦 传统渲染 vs 全局光照

```
传统渲染（无 GI）               有全局光照（GI）
┌──────────────┐               ┌──────────────┐
│              │               │              │
│  ☀️           │               │  ☀️           │
│   │          │               │   │          │
│   ▼          │               │   ▼          │
│  🪑 亮        │               │  🪑 亮        │
│  🪈 暗 ❌     │               │  🪈 微亮 ✅   │
│  🚪背后 全黑  │               │  🚪背后 微亮  │
│              │               │              │
└──────────────┘               └──────────────┘
```

**问题是**：真正的全局光照（比如电影级渲染用的路径追踪）**每帧要算几百万条光线的多次弹射**，一帧就要渲染几小时——游戏要跑 60 帧/秒，根本做不到。

**NVIDIA 的解决方案：RTXGI-DDGI（动态漫反射全局光照）**

> DDGI = Dynamic Diffuse Global Illumination（动态漫反射全局光照）
>
> 它不是算每条光线的精确路径，而是**在场景中放置一批"探针"，用探针来估算间接光照**——又快又够好。

---

## 二、用故事理解 DDGI：蜜蜂采蜜

想象你是一个**养蜂人**，想知道你的**大花园里每个角落的花粉分布情况**：

### 🐝 传统方法（路径追踪）
- 你派出 1000 只蜜蜂
- 每只蜜蜂从蜂巢出发，在花园里**随机乱飞**
- 最终回来告诉你"我去了哪里、采了多少花粉"
- ✅ 结果非常精确
- ❌ 但蜜蜂飞得太慢了——等它们回来，花都谢了

### 🐝 DDGI 方法（探针法）
你在花园里**插了 100 根探针**（就像温度计），每根探针相当于一个小型传感器：

```
🌳 大花园
┌─────────────────────────────────┐
│                                 │
│  📡     📡     📡     📡       │
│       🌸                        │
│  📡  🌸  📡     📡     📡     │
│                                 │
│  📡     📡  🌸  📡     📡     │
│       🏠                        │
│  📡     📡     📡     📡       │
│                                 │
└─────────────────────────────────┘
```

**每根探针的工作：**
1. 每隔一段时间，**向四面八方发射光线**（采样）
2. 记录"这个方向有多亮"（辐照度）
3. 把结果存在探针里
4. 当需要知道某个位置的光照时→看附近的探针→**插值计算**

**关键优势：**
- 探针之间可以**共享数据**（100 根探针覆盖整个花园）
- 光线只在**探针位置**发射，而不是每个像素
- 计算量大大减少，**适合实时渲染**

## 三、DDGI 的整体架构

DDGI 就像一个**四条腿的桌子**，每条腿负责一个阶段：

```
DDGI 渲染管线（4 大阶段）
┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  ① GBuffer 生成   ② 探针更新       ③ 间接采样     ④ 合成输出  │
│  (GBufferRGS)     (Probe Update)  (Sample Irr.)  (Composite) │
│                                                              │
│  只算直接光照      更新探针数据     从探针查间接光   混合所有光照  │
│  存到 GBuffer     5 个子步骤       最终渲染画面     颜色校正     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

### 每条腿的职责

| 阶段 | 本质工作 | 占用时间 | 就像 |
|------|---------|---------|------|
| **① GBuffer 生成** | 算出每个像素的"基础信息"（位置、法线、颜色） | 快 | 给每个学生发一张答题卡 |
| **② 探针更新** | 更新场景中探针的光照数据 | 中等 | 老师批改试卷 |
| **③ 间接采样** | 用探针数据算间接光照 | 快 | 学生看成绩单算总分 |
| **④ 合成输出** | 合成直接光+间接光+后处理 | 快 | 打印最终成绩单 |

## 四、第一阶段：GBuffer 生成（GBufferRGS）

### 4.1 什么是 GBuffer？

GBuffer = Geometry Buffer（几何缓冲区）。它是一个**"信息集装箱"**，存储了每个像素的：

| 缓冲区 | 存储内容 | 就像 |
|--------|---------|------|
| GBuffer A | 表面颜色（Albedo）+ 材质类型标记 | 物体的"本色" |
| GBuffer B | 世界空间位置 + 光线击中距离 | 物体"在哪" |
| GBuffer C | 表面法线（编码在 [0,1]） | 物体"朝向哪" |
| GBuffer D | 直接光照结果（可选） | 太阳直接照到的亮度 |

### 4.2 工作流程

```
对于屏幕上的每一个像素：
  │
  ├── 从相机发射一条光线
  ├── 光线撞到物体 → 记录交点信息（位置、法线、材质颜色）
  ├── 计算直接光照（光源到物体，不弹射）
  ├── 把信息存入 GBuffer A/B/C/D
  │
  └── 没撞到任何物体 → 标记为"天空"或"背景"
```

> **重要：这一阶段只算直接光照，不算间接光照。** 间接光照留给后面的阶段处理。

### 4.3 和路径追踪的对比

| 对比项 | GBufferRGS（DDGI用） | PathTraceRGS（路径追踪） |
|--------|---------------------|----------------------|
| 🎯 **目标** | 生成 GBuffer 数据，给后处理用 | 直接渲染最终画面 |
| 🔢 **光线数量** | 每个像素 **1 条**光线 | 每个像素 **几十到几百条** |
| 🔄 **弹射次数** | **无弹射**（只算直接光） | **多次弹射**（模拟间接光） |
| ⏱️ **速度** | ✅ 快，适合实时 | ❌ 慢，适合离线 |
| 📊 **结果** | 中间数据（GBuffer） | 最终画面 |

> **简单说：** GBufferRGS 是"粗加工"，PathTraceRGS 是"精加工"。DDGI 用粗加工 + 探针技巧，假装自己有精加工的效果。

## 五、第二阶段：探针更新（DDGI 的核心）

这是 DDGI **最重要也最复杂**的阶段。它由 **5 个子步骤**组成，就像一个流水线：

### 5.1 全局图

```
探针更新流水线
┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐   ┌──────────┐
│ ① 光线   │ → │ ② 数据   │ → │ ③ 重新   │ → │ ④ 分类   │ → │ ⑤ 变化   │
│ 追踪     │   │ 更新     │   │ 定位     │   │         │   │ 计算     │
│          │   │          │   │          │   │          │   │          │
│ Probe    │   │ Update   │   │ Relocate │   │ Classify │   │ Calc     │
│ Raytrace │   │ Probes   │   │ Probes   │   │ Probes   │   │Variability│
└──────────┘   └──────────┘   └──────────┘   └──────────┘   └──────────┘
```

### 5.2 逐个步骤说人话

#### ① 光线追踪（ProbeRaytrace）

```
对于每个需要更新的探针：
  │
  ├── 向四面八方发射 N 条光线（比如 64 条）
  │    方向分布：均匀分布在半球空间上
  │    （可以用 Hammersley 序列减少采样数）
  │
  ├── 每条光线撞到物体：
  │     ├── 记录"撞到哪了"（位置、距离）
  │     └── 计算"那里有多亮"（直接光照 + BRDF）
  │
  └── 没撞到物体：
        └── 记录"环境光"的亮度
```

**就像：** 探针伸出 N 根触手去摸周围的环境，每根触手报告"我碰到了什么、有多亮"。

#### ② 数据更新（UpdateProbes）

```
对于每个探针：
  │
  ├── 辐照度更新：把 N 条光线采集到的亮度平均
  │    → 得到探针位置的"平均间接光照"
  │
  └── 距离更新：记录每条光线飞了多远才撞到物体
      → 用于后续的"可见性测试"
```

**就像：** 探针把触手采集到的信息汇总，记在本子上。

#### ③ 重新定位（RelocateProbes）

```
对于每个探针：
  │
  ├── 检查：探针是不是卡在物体内部了？
  │    ↓ 是
  └── 把探针沿着法线方向推出去，放到表面外面
```

> **为什么要做这个？** 探针每隔一段时间就要检查自己是不是"卡墙里"了。如果是，就自己挪出来。

**就像：** 如果你把温度计插进了墙壁，它测的是墙内的温度而不是房间的温度——你需要把它拔出来一点。

#### ④ 分类（ClassifyProbes）

```
对于每个探针：
  │
  ├── 计算：当前帧的亮度和上一帧的亮度差多少？
  │    ↓ 差很多
  │    → 标记为"需要更新"（活跃）
  │    ↓ 差很少
  │    → 标记为"不需要更新"（静止）
  │
  └── 下一帧只更新"活跃"探针，跳过"静止"探针
```

> **为什么要做这个？** 场景大部分区域是静止的——墙壁不会突然变亮。只更新变化的部分，**省掉 80% 的计算量**。

**就像：** 教室里大部分学生都在安静做题，你只需要关注那几个在偷偷玩手机的学生。

#### ⑤ 变化计算（CalcVariability）

```
对于每个探针：
  │
  ├── 分析历史亮度变化趋势
  │    （过去几帧的亮度变化幅度）
  │
  ├── 变化大 → 高频更新（甚至每帧都更新）
  │
  └── 变化小 → 低频更新（几帧才更新一次）
```

**就像：** 一个不停在跳舞的人（高变化），你需要一直盯着他；一个坐着看书的人（低变化），你瞥一眼就够了。

## 六、第三阶段：间接采样（SampleIrradiance）

探针更新完后，它们已经记录了场景中每个区域的间接光照数据。现在需要**用这些数据来渲染画面**。

### 6.1 工作流程

```
对于屏幕上的每个像素：
  │
  ├── ① 从 GBuffer 取出这个像素的位置和法线
  │
  ├── ② 在 3D 空间中，找到最近的 8 个探针
  │    （就像在一个立方体的 8 个顶点上各有一个探针）
  │
  ├── ③ 对每个探针：
  │     ├── 取探针的辐照度值（亮度）
  │     ├── 计算可见性权重（探针和像素之间有没有遮挡？）
  │     │    └── 用 Chebyshev 不等式近似判断
  │     └── 乘以插值权重（距离越近权重越大）
  │
  └── ④ 将所有探针的贡献加起来 → 得到这个像素的间接光照
```

### 6.2 可见性测试（关键技巧）

这是 DDGI 最巧妙的优化之一。

**问题：** 像素 A 在一个房间，探针 B 在隔壁房间。虽然距离很近，但中间隔了一堵墙，探针 B 的光照不应该影响像素 A。

**传统做法：** 再发一条光线验证中间有没有墙 → 又慢又贵

**DDGI 的做法（Chebyshev 不等式）：**

```
像素到探针的距离 = d
探针记录的所有光线击中距离 = {h₁, h₂, ..., hₙ}
                    均值 = μ，方差 = σ²

如果 d > μ（像素比探针的平均"视野"更远）：
  → 大概率有遮挡 → 降低权重
  具体公式：P(d > μ) ≈ σ² / (σ² + (d-μ)²)

如果 d ≤ μ（像素在探针的"视野"内）：
  → 大概率无遮挡 → 权重不变
```

> **就是不用发射光线，用统计学判断"有没有被挡住"。** 这是 DDGI 能实时运行的核心技巧之一。

## 七、第四阶段：合成输出（DDGIComposite）

最后一步，把所有光照合成最终画面：

```
对于每个像素：
  │
  ├── 获取直接光照（从 GBuffer D 或者单独计算）
  │
  ├── 获取间接光照（从 DDGI 采样阶段）
  │
  ├── （可选）获取环境光遮蔽（RTAO）
  │
  ├── 加起来 → 总光照
  │
  └── 后处理：
       ├── 曝光控制（Tone Mapping）
       ├── 颜色映射（如 ACES Filmic Curve）
       └── Gamma 校正（sRGB）
```

**最后输出到屏幕 → 你就看到了一幅有真实全局光照的画面！**

## 八、进阶优化技巧

### 8.1 渐进式累积（Progressive Accumulation）

```
帧 1：每个像素发射 1 条光线 → 画面很噪
帧 2：再发 1 条 → 和上次结果平均 → 噪音减少
帧 3：再发 1 条 → 平均 → 噪声更少
...
帧 N：N 条光线平均 → 画面干净
```

> **就是"每一帧只算一点，慢慢积累"。** 游戏运行时你看到的第一帧可能有点噪，但几帧之后画面就变干净了。

**就像：** 你每秒钟拍一张照片，然后把所有照片叠加平均——噪点被"平均掉"了。

### 8.2 SER（Shader Execution Reordering）

这是 NVIDIA 的独门优化。

**问题：** GPU 一次处理 32 个像素（一个 warp）。如果有的像素走了"反射"路径、有的走了"折射"路径、有的走了"漫反射"路径——它们就**分岔**了（divergence），GPU 的效率大幅下降。

**SER 的解决方案：**

```
传统方式：           SER 优化后：
┌──────┐           ┌──────┐
│ 像素1 ──反射     │ 所有反射像素 → 一起算 ✅
│ 像素2 ──折射     │ 所有折射像素 → 一起算 ✅
│ 像素3 ──反射     │ 所有漫反射像素 → 一起算 ✅
│ 像素4 ──漫反射   │
│ 像素5 ──折射     │ 效率提升 2~3 倍！
│ 像素6 ──反射     │
└──────┘           └──────┘
  混乱 → 效率低        分类 → 效率高
```

> **就像把快递按小区分类再配送**，而不是每个快递员都跑遍全城。

### 8.3 Bindless 资源（无绑定资源）

传统方式下，Shader 要访问一个纹理，需要提前"绑定"到固定的寄存器位置。

```
传统绑定：
  纹理 A → 寄存器 t0
  纹理 B → 寄存器 t1
  ...
  最多只能同时访问几十个纹理

Bindless：
  RWTexture2D<float4> RWTex2D[] : register(u6, space0);
  通过索引动态访问：RWTex2D[任意ID]
  可以同时访问成千上万个纹理！
```

> DDGI 的探针需要访问大量数据（辐照度、距离、状态等），Bindless 让 Shader 可以动态选择资源，**不需要每帧重新绑定**。

## 九、DDGI vs 路径追踪

| 对比项 | DDGI（实时） | 路径追踪（离线） |
|--------|------------|----------------|
| 🎯 **目标** | 游戏 60fps | 电影级渲染 |
| ⚡ **光线数量** | 探针位置发射少量光线 | 每个像素发射数千条光线 |
| 🏗️ **间接光照** | 探针插值（近似） | 精确物理模拟 |
| 🪞 **焦散效果** | ❌ 不支持 | ✅ 完美支持 |
| 🔄 **多次弹射** | 🟡 有限 | ✅ 任意次数 |
| 📺 **硬件要求** | RTX 显卡 | CPU/GPU 均可 |
| ⏱️ **速度** | ✅ 实时（<16ms） | ❌ 几分钟到几小时 |

**一句话选哪个：**

> **做游戏 → DDGI**（实时、够用、性价比高）
>
> **做电影 → 路径追踪**（精确、真实、需要时间）

## 十一、总结

### 一句话说清 DDGI

> **DDGI = 在场景中放一批"光照探针" → 探针采样环境光照 → 渲染时查探针估算间接光照 → 又快又够好地模拟全局光照。**

### DDGI 的贡献

| 贡献 | 为什么重要 |
|------|-----------|
| ✅ **探针机制** | 用少量探针代替全分辨率光照计算，大幅降低计算量 |
| ✅ **增量更新** | 只更新变化的部分，静止区域几乎不花计算资源 |
| ✅ **Chebyshev 可见性** | 公式代替光线追踪，零额外光线开销做可见性判断 |
| ✅ **并行架构** | 探针更新和主渲染并行执行，不阻塞管线 |
| ✅ **SER + Bindless** | 利用 NVIDIA 硬件优化，进一步提升效率 |

### 和 DSEngine 的关系

DSEngine 目前使用 **Forward 光栅化渲染**（`ForwardScenePass` + PreZ + Shadow + Bloom + SSAO + FXAA + Composite 等 Pass），搭配 Shadow Mapping 和 PBR 着色器。如果未来计划实现更真实的间接光照效果，DDGI 是一个非常有价值的研究方向：

- DSEngine 有 **Vulkan 后端**，但**暂未集成 Ray Tracing 扩展**（如 `VK_KHR_ray_tracing`），需要先扩展 RHI 层
- DDGI **不是简单的后处理模块**——它需要重做渲染管线（探针更新 + 间接采样 + 合成），属于深度集成
- 现阶段更可行的第一步是：在**离线烘焙管线**中实现简单的**光照探针（Light Probe）**系统，用 CPU 预计算间接光照，运行时查表
- 现阶段可以先在离线烘焙管线中尝试简单的**光照探针**系统

## 十二、路径追踪模式（Path Tracing）的完整解析

DDGI 用探针实现了**实时**全局光照，但牺牲了精度。如果计算机有足够的时间（比如离线渲染），可以用**路径追踪**来获得更真实的效果。

### 12.1 路径追踪的核心思想

```
对于每个像素：
  │
  ├── 发射 N 条"主射线"（抗锯齿采样）
  │
  └── 每条主射线都会经历"多次反弹"：
        ├── 第 1 次：直接光照（光源→物体→眼睛）
        ├── 第 2 次：一次间接光照（光源→A→B→眼睛）
        ├── 第 3 次：二次间接光照（光源→A→B→C→眼睛）
        └── …直到能量衰减到忽略不计
```

**和 DDGI 的关键区别：**

| 对比 | DDGI | 路径追踪 |
|------|------|---------|
| 间接光照来源 | **查探针数据**（近似） | **直接模拟物理**（精确） |
| 光线跟踪 | 探针发光线，像素不发光线 | **每个像素都发很多光线** |
| 多次反弹 | 探针间接模拟 | **显式追踪每一次反弹** |

### 12.2 PathTraceRGS：路径追踪入口着色器

PathTraceRGS 是路径追踪模式的入口，它负责控制**每个像素的路径追踪主循环**。

**输入与输出：**
- 输入：像素坐标、场景加速结构（TLAS）、随机数种子、路径追踪参数
- 参数：`PTSamplesPerPixel`（每像素采样数）、`PTNumBounces`（最大反弹次数）、`PTMaxThroughput`（最大吞吐量）
- 输出：最终像素颜色

**执行流程：**

```
对于每个像素：
  │
  ├── 初始化颜色 = 0
  │
  ├── 对于每个样本（共 PTSamplesPerPixel 次）：
  │     ├── 生成主射线（带随机偏移，实现抗锯齿）
  │     ├── 初始化路径状态（起点=相机，方向=主射线方向）
  │     │     └── 吞吐量 = 1.0（路径能量初始值）
  │     │
  │     └── 反弹循环（最多 PTNumBounces 次）：
  │           ├── TraceRay() 追踪射线
  │           ├── 如果命中：
  │           │     ├── 读取表面属性（位置、法线、反照率等）
  │           │     ├── 如果命中光源 → 累加颜色
  │           │     ├── 计算直接光照贡献 → ×吞吐量 → 累加
  │           │     ├── 更新吞吐量（反照率 × 衰减因子）
  │           │     ├── 如果吞吐量 < 阈值 → 提前终止
  │           │     └── 生成下一射线方向（BRDF 采样）
  │           │
  │           └── 如果未命中：
  │                 └── 环境光 × 吞吐量 → 累加
  │
  └── 输出 = 总颜色 / 样本数
```

### 12.3 可视化对比：DDGI vs 路径追踪的每条光线

```
DDGI 模式（4 步）：
┌─────────┐   ┌───────────┐   ┌──────────┐   ┌────────┐
│ GBuffer │   │ 探针更新   │   │ 间接采样  │   │ 合成   │
│ 1 光线/像素│   │ 64光线/探针 │   │ 查探针    │   │ 后处理  │
└─────────┘   └───────────┘   └──────────┘   └────────┘

路径追踪模式（1 步）：
┌──────────────────────────────────────────────────────┐
│ PathTraceRGS                                          │
│ 像素1: ──↘──↘──↘──↘ (4次反弹) → 颜色                  │
│ 像素2: ──↘──↘──↘    (3次反弹) → 颜色                  │
│ 像素3: ──↘──↘──↘──↘──↘ (5次反弹) → 颜色               │
│ ...                                                    │
│ 每个像素 N 条主射线 × M 次反弹                          │
└──────────────────────────────────────────────────────┘
```

**关键点：** 路径追踪的`TracePath`函数将单次路径追踪封装起来，可以递归调用自身（或用循环模拟递归），每次调用处理一次反弹。

## 十三、直接光照计算：DirectDiffuseLighting

无论是 DDGI 还是路径追踪，都需要计算**直接光照**（光源直接照射到物体表面的光）。文章中的 `DirectDiffuseLighting` 函数专门做这件事。

### 13.1 核心思路

```
对于表面上的一个点 P：

遍历场景中的每个光源：
  ├── 方向光（太阳）：阴影射线检测 → 法线·光线方向 × 光源强度
  ├── 点光源（灯泡）：阴影射线检测 → 距离平方衰减 × 法线·光线方向
  └── 聚光灯（手电筒）：阴影射线检测 → 角度衰减 × 距离衰减 × 方向衰减

所有光源贡献相加 → 乘以漫反射 BRDF（反照率 / π）→ 最终结果
```

### 13.2 阴影射线（Shadow Ray）的机制

阴影射线是判断"光源是否被遮挡"的关键技术：

```
表面点 P

对每个光源：
  │
  ├── 从 P 向光源方向发射一条"阴影射线"（Shadow Ray）
  ├── 射线最大长度 = P 到光源的距离
  │
  ├── 如果射线没有碰到任何物体 → 光源可见 → 贡献=100%
  │
  └── 如果射线碰到了物体 → 光源被遮挡 → 贡献=0%（硬阴影）
```

> Shadow Ray 是**光照计算的主要性能开销**。DDGI 用探针 + Chebyshev 不等式避免了对每个像素都发阴影射线，这是它能实时运行的关键。

### 13.3 三种光照的计算细节

| 光源类型 | 计算内容 | 衰减公式 |
|---------|---------|---------|
| ☀️ **方向光** | 太阳等无限远光源，方向处处相同 | 无距离衰减，只有 `max(dot(N, L), 0)` |
| 💡 **点光源** | 灯泡等从一点向四周发光 | `1 / distance²` 平方反比衰减 |
| 🔦 **聚光灯** | 有方向限制的光源，形成光锥 | 平方反比 × 角度衰减（SpotAttenuation） |

## 十四、资源管理：Bindless（无绑定）技术

DDGI 和路径追踪需要访问大量纹理和缓冲区（每个探针的辐照度纹理、每个像素的颜色缓冲区等）。传统方式需要预先"绑定"到固定寄存器，但这里**数量太大**。

### 14.1 传统绑定 vs Bindless

```
传统方式（OpenGL/D3D11）：
  纹理 A → 寄存器 t0
  纹理 B → 寄存器 t1
  纹理 C → 寄存器 t2
  ...
  限制：同时只能绑定几十个

Bindless（Vulkan/D3D12）：
  RWTexture2D<float4> RWTex2D[] : register(u6, space0);
  通过索引访问：RWTex2D[任意ID]
  优势：同时访问成千上万个！
```

**就像：**
- 传统方式：你只能同时看面前的 10 本书（桌面有限）
- Bindless：你有一个巨型图书馆，想看哪本就看哪本（按索引取书）

### 14.2 在 DDGI 中的应用

DDGI 有大量探针数据需要动态访问：
- 每个探针的**辐照度纹理**（亮度数据）
- 每个探针的**距离缓冲区**（可见性数据）
- 每个探针的**状态标记**

用 Bindless，着色器可以这样写：

```hlsl
// 按探针索引动态访问
float3 irradiance = RWTex2D[probeIndex].xyz;
float distance = RWTex2DDistance[probeIndex].x;
```

**不需要每帧重新绑定描述符，大幅降低 CPU 开销。**

## 十五、SER（Shader Execution Reordering）—— NVIDIA 的独门优化

### 15.1 为什么需要 SER？

GPU 一次处理 32 个像素（一个 warp）。**如果这 32 个像素走了不同的代码路径**——比如有的进反射着色器、有的进折射着色器、有的进漫反射着色器——就会出现"线程分歧"：

```
没有 SER：
  Warp 中的 32 个线程：
  [反射][折射][反射][漫反射][折射][反射]...
  → GPU 必须"串行化"执行不同路径
  → 效率大幅下降

有 SER：
  Warp 被重新排序：
  [反射][反射][反射][折射][折射][漫反射]...
  → 同类路径连续执行
  → 效率提升 2~3 倍
```

### 15.2 SER 的工作原理

SER 把"光线追踪"和"命中着色器执行"分离开来：

```
传统方式：
  TraceRay()  →  立即执行命中着色器
  
SER 方式：
  NvTraceRayHitObject()  →  收集所有命中信息
       ↓
  NvReorderThread()      →  对线程重排序
       ↓
  NvInvokeHitObject()    →  批量执行命中着色器
```

> **就像快递分拣：** 不 SER 时每个快递小哥随机送货（满城跑）；有 SER 时先把所有快递按小区分类，再集中配送（效率极高）。

## 十六、GBufferRGS vs PathTraceRGS 完整对比

这是文章中一个重要的对比表格，用 5 个维度说明两个着色器的根本区别：

| 对比维度 | GBufferRGS | PathTraceRGS |
|---------|-----------|-------------|
| 🎯 **渲染目标** | **生成 GBuffer**，为后续光照处理提供数据 | **输出最终颜色**，直接完成渲染 |
| 🔢 **算法复杂度** | 每像素 **1 条主射线**，无反弹 | 每像素 **N 条射线 × M 次反弹**，复杂度高几个数量级 |
| 💡 **光照计算** | **仅直接光照** | **直接 + 间接光照**，含多次反弹和能量衰减 |
| 📊 **输出处理** | 输出到多个 GBuffer 纹理 | 输出最终像素颜色（含色调映射、Gamma 校正） |
| ⏱️ **性能** | ✅ **高**，适合实时渲染 | ❌ **低**，适合离线/参考渲染 |

### 用工程类比

> **GBufferRGS = 施工现场的测量员**
> 只测量"哪里有墙、墙是什么颜色"，不装修。
>
> **PathTraceRGS = 顶级室内设计师**
> 亲自挑选每一种材料、搭配每一件家具，做出成品。
>
> **DDGI = 一个有经验的工头**
> 看一眼测量员的图纸，凭经验估算"这里应该怎么装修"——比设计师快 100 倍，效果也有 80% 的水准。

## 总结

### DDGI 的核心竞争力

> **DDGI 不是要替代路径追踪，而是在"实时"和"真实"之间找到了最佳的平衡点。**
>
> - 需要**交互式预览**或**游戏运行时** → 用 DDGI（探针≈足够好）
> - 需要**最终输出**或**电影级质量** → 用路径追踪（精确≈最好）
> - 两者可以**共存于同一个引擎**中，DDGI 做实时预览，路径追踪做离线烘焙

### 与 DSEngine 的关系

DSEngine 的渲染管线目前基于 **Forward 光栅化**（`ForwardScenePass`）+ PBR + Shadow Mapping。如果未来需要增强全局光照：

| 接入方式 | 复杂度 | 效果 |
|---------|--------|------|
| 先接入**一级间接光照探针**（简化版 DDGI） | 🟡 中 | 显著提升室内场景真实感（需先扩展 RHI 层支持 Ray Tracing） |
| 完整接入 **Vulkan Ray Tracing + DDGI** | 🔴 高 | 照片级的实时全局光照（需先集成 `VK_KHR_ray_tracing` 扩展） |
| 离线**路径追踪烘焙**光照贴图 | 🟡 中 | 静态场景接近电影级质量（可用 CPU 路径追踪库实现） |
