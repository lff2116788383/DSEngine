# JobSystem——怎么让 CPU 所有核心一起干活？

> **灵魂拷问：你的电脑是 8 核 16 线程，但为什么有时候打游戏还是会卡？**
>
> 因为**不是核心越多就越快的**。如果程序写得不好，16 个核心里可能只有 1 个在干活，另外 15 个在旁边看着。JobSystem 就是来解决这个问题的——它像一个聪明的包工头，把活分给所有工人，一个都不闲着。

## 一、单线程的问题：一个工人干所有活

### 早期的游戏只有一个线程

在 2000 年代初，大部分游戏都是**单线程**的——不管你的 CPU 有几个核心，游戏只用一个。

```
单线程模式：
  ┌─────────────────────────────────────────┐
  │  CPU 核心 1：                             │
  │  处理输入 → 更新逻辑 → 物理模拟          │
  │  → AI 计算 → 渲染准备 → 等待 GPU         │
  │  → 处理输入 → 更新逻辑 → ...             │
  │                                          │
  │  CPU 核心 2：                             │
  │  [ 闲 置 ]                               │
  │                                          │
  │  CPU 核心 3：                             │
  │  [ 闲 置 ]                               │
  │                                          │
  │  CPU 核心 4：                             │
  │  [ 闲 置 ]                               │
  └─────────────────────────────────────────┘
```

### 为什么早期游戏这样做？

因为那时候 CPU 只有 1 个核心。多线程编程又非常复杂，程序员觉得：**"一个核心够用了，搞多线程容易出 Bug。"**

但现在一个入门级 CPU 都有 4 个核心，高端的有 16 甚至 32 个核心。如果还用单线程：

> **花了 1000 块钱买了 8 个核心，实际只用到了 1 个核心——也就是 125 块钱的效果。另外 875 块钱白花了。**

### 用生活场景理解

📦 **搬家公司：**

你搬家了，叫了搬家公司。

**单线程模式（一个工人）：**
```
你叫了 8 个工人（8 核 CPU）
但只让 1 个工人干活：
  
  工人 1：搬沙发 → 搬床 → 搬桌子 → 搬箱子 → 搬电视 → …
  工人 2~8：[ 坐 在 卡 车 上 看 ]
  
  结果：搬了 8 个小时才搬完
        7 个工人白拿工资
```

这就是单线程的问题：**花了 8 个人的钱，只用到 1 个人。**

## 二、多线程的问题：分工难、抢资源、死锁

### 手动多线程

既然一个核心不够，那就用多个核心——直接开多个线程。

但这并不简单。想象一下：

```
多线程模式（手动管理）：
  线程 1：处理物理碰撞
  线程 2：处理 AI 逻辑  
  线程 3：处理动画更新
  线程 4：处理网络数据
  
  听起来不错，但问题来了……
```

### 问题 1：分工不均

```
线程 1（物理）：忙得要死，要算 10000 个碰撞检测
线程 2（AI）：闲得发慌，就 3 个敌人，0.1 秒算完了
线程 3（动画）：中等忙碌
线程 4（网络）：大部分时间在等待网络数据

结果：线程 2 在摸鱼，线程 1 在拼命
      整体性能被最慢的那个线程拖着
```

### 问题 2：抢资源（锁竞争）

两个线程同时访问同一个数据，程序会崩溃。所以要用"锁"来保护数据：

```
线程 1：我要修改"玩家血量"
        → 上锁 → 修改中...
线程 2：我也要修改"玩家血量"
        → 等锁 → 等锁 → 等锁...
        → 线程 1 解锁了 → 终于轮到我
        
两个线程不是在干活，而是在"等锁"
```

更糟糕的情况：**两个线程互相等对方解锁，永远等下去——这叫"死锁"。**

```
线程 1：锁了 A，等 B
线程 2：锁了 B，等 A

线程 1：B 什么时候解锁？
线程 2：A 什么时候解锁？
——两个线程永远卡死在这里
```

### 用生活场景理解

🚗 **只有一个厕所的办公室：**

你在一栋写字楼里上班，每层 100 人，但每层只有 1 个厕所。

```
早上 9 点：
  100 个人同时想上厕所
  → 厕所门口排长队（锁竞争）
  → 有人等不及走了（性能浪费）
  → 有人敲门问"好了没"（轮询）
  
更可怕的情况：
  两个人各进一个隔间，
  互相堵着门出不来
  → 死锁
```

### 手动多线程的问题总结

| 问题 | 说明 | 后果 |
|:----|:------|:-----|
| ⚖️ **负载不均** | 有的线程忙死，有的闲死 | 总性能被最慢的线程限制 |
| 🔒 **锁竞争** | 多个线程抢同一个资源 | 大部分时间在等待 |
| 💀 **死锁** | 互相等对方解锁 | 程序卡死 |
| 😱 **数据竞争** | 多个线程同时改数据 | 程序崩溃，Bug 极难复现 |
| 🤯 **复杂度** | 要手动管理线程的创建、销毁、同步 | 程序员写秃头 |

## 三、JobSystem：把大任务拆成小任务，自动分配给空闲核心

### JobSystem 的核心思想

> **别让程序员手动分配任务给线程。让程序员只管"拆任务"，系统自动把任务分配给空闲的 CPU 核心。**

```
传统多线程：
  程序员：线程 1 做物理，线程 2 做 AI，线程 3 做动画……
  
JobSystem：
  程序员：我把物理拆成 100 个小任务放进去
          AI 拆成 20 个小任务放进去
          动画拆成 50 个小任务放进去
          谁来干？JobSystem 你自己分配
```

### JobSystem 怎么工作

```
               任务队列（Job Queue）
  ┌──────────────────────────────────────────┐
  │  [物理-01] [物理-02] [AI-01] [动画-01]   │
  │  [物理-03] [物理-04] [AI-02] [动画-02]   │
  │  [物理-05] [...]        [...]   [...]    │
  └──────────────────────────────────────────┘
               ↓    ↓    ↓    ↓
        ┌──────┐ ┌──┐ ┌──┐ ┌──┐
        │核心1 │ │核心│ │核心│ │核心│
        │      │ │ 2 │ │ 3 │ │ 4 │
        └──────┘ └──┘ └──┘ └──┘
        
  每个核心空闲了，就去任务队列拿一个任务做
  做完再拿下一个，直到所有任务完成
```

### 用生活场景理解

🍕 **披萨店接大订单：**

你开了一家披萨店，来了一个 100 人的大团餐。

**手动分配（传统多线程）：**
```
厨师长：小张你做 25 个，小李你做 25 个，
        小王你做 25 个，小赵你做 25 个。
        
小张做了 20 个，手受伤了，做得很慢
小李做得快，25 个做完开始摸鱼
小王做得一般
小赵也在摸鱼

结果：等小张做完，其他人都闲着
```

**JobSystem 模式：**
```
厨师长：我把 100 个披萨的单子放进任务箱
        谁有空谁来拿

小张做得慢 → 只拿了 15 个任务
小李做得快 → 做完 25 个，又去拿小张没做完的
小赵也不闲着 → 继续拿任务做

结果：所有人都在干活，没有一个人摸鱼
      整体完成时间大大缩短
```

### 伪代码

```cpp
// 定义一个"任务"
struct Job {
    void (*function)(void* data);  // 要执行的函数
    void* data;                     // 函数需要的参数
};

// JobSystem 核心
class JobSystem {
private:
    // 线程安全的任务队列
    std::queue<Job> job_queue_;
    // 所有工作线程
    std::vector<std::thread> workers_;
    
public:
    // 添加一个任务
    void addJob(Job job) {
        lock();
        job_queue_.push(job);  // 把任务放进队列
        unlock();
        notifyOneWorker();     // 通知一个空闲线程：有新任务了！
    }
    
    // 工作线程的主循环
    void workerMain() {
        while (true) {
            Job job = waitForJob();  // 等任务
            job.function(job.data);  // 做任务
            // 做完继续等下一个
        }
    }
};
```

## 四、工作窃取：你的活干完了？去帮别人干

### 为什么需要工作窃取？

上面的 JobSystem 有一个问题：**所有线程共享一个任务队列。**

```
一个队列的问题：
  所有线程都要从同一个队列拿任务
  → 每次拿任务都要"上锁"
  → 线程越多，锁竞争越严重
  
  16 个线程抢一个队列的任务：
  大部分时间不是在干活，而是在抢任务
```

### 工作窃取（Work Stealing）的思路

> **每个线程有自己的任务队列。先干自己的活，干完了再去别人的队列"偷"任务。**

```
工作窃取模式：

  线程 1 的队列：     线程 2 的队列：    线程 3 的队列：    线程 4 的队列：
  [任务] [任务]        [任务]              [任务] [任务]      [空]
  [任务] [任务]                              [任务]
  
  线程 1：做自己的任务
  线程 2：做完自己的 → 看看线程 1 的队列？
           → 偷一个任务过来做
  线程 3：做自己的任务
  线程 4：自己的队列空了 → 看看线程 3 的队列？
           → 偷一个任务过来做
```

### 为什么工作窃取效率高？

```
每个线程操作自己的队列 → 不需要锁（或者很少需要锁）

只有"偷"别人的任务时才需要锁
→ 但"偷"发生的频率低得多
→ 锁竞争大大减少

结果：
    16 个线程，16 个队列
    大部分时间各干各的
    偶尔有人偷任务
    锁竞争趋近于零
```

### 用生活场景理解

🏭 **快递分拣中心：**

每个分拣员面前有一条传送带。

```
不偷的模式：
  - 所有包裹从一条大传送带过来
  - 10 个分拣员抢着拿
  - 经常撞在一起

工作窃取模式：
  - 每个分拣员有自己的小传送带
  - 自己的包裹自己分拣
  - 分完了？看看旁边同事忙不忙
  - 同事忙 → 帮他分几个
  - 同事也忙 → 继续找下一个
```

### 工作窃取 vs 普通队列

| 对比 | 一个共享队列 | 工作窃取（每个线程独立队列） |
|:----|:-----------|:-------------------------|
| **拿任务** | 要上锁 | 基本不用锁 |
| **锁竞争** | 严重（线程越多越严重） | 轻微（只有偷时才锁） |
| **负载均衡** | 自动的 | 通过"偷"来均衡 |
| **实现复杂度** | 简单 | 较复杂 |
| **CPU 缓存友好** | ❌ 任务和线程不绑定 | ✅ 同一个线程反复做类似任务 |

## 五、生活中的类比：快递分拣中心、搬家公司

### 类比 1：快递分拣中心 📦

| 现实世界 | JobSystem |
|:--------|:---------|
| 一大车快递到了 | 一帧开始，一大批任务需要处理 |
| 拆分成一个个包裹 | 拆分成一个个 Job |
| 传送带把包裹送到分拣员面前 | 任务队列 |
| 分拣员拿包裹扫码分拣 | 工作线程执行任务 |
| 自己的活干完了，帮旁边的同事 | 工作窃取 |
| 所有包裹分拣完，休息等下一车 | 同步等待，准备下一帧 |

### 类比 2：搬家公司 🚚

不用"一个工人干所有活"（单线程），也不用"提前分配谁搬什么"（手动多线程），而是：

> **所有工人站在卡车旁边，每个人搬得动什么就搬什么。搬完一个回去再搬下一个。有人搬得快，就多搬几个；有人搬得慢，就少搬几个。大家都不闲着，活干得最快。**

### 类比 3：自助餐厅 🍽️

| 场景 | 对应 |
|:----|:-----|
| 自助餐厅的取餐台 | 任务队列 |
| 取餐的食客 | 工作线程 |
| 每个人按需取餐 | 任务分配 |
| 某样菜没了，去吃别的 | 没有对应任务，线程空闲 |
| 好朋友帮你拿一份 | 工作窃取 |

## 六、对比表格：单线程 vs 多线程 vs JobSystem

| 对比维度 | 单线程 | 手动多线程 | JobSystem |
|:--------|:------|:----------|:---------|
| **CPU 利用率** | ❌ 只用一个核心 | ⚠️ 看运气，可能分布不均 | ✅ 所有核心满载 |
| **编程难度** | ✅ 最简单 | ❌ 极难（锁、死锁、竞争） | ✅ 只需要拆任务 |
| **负载均衡** | ✅ 不用考虑 | ❌ 手动分配很难均衡 | ✅ 自动分配 + 工作窃取 |
| **锁竞争** | ✅ 没有锁 | ❌ 严重 | ✅ 很少（工作窃取） |
| **扩展性** | ❌ 换更多核没提升 | ⚠️ 需要重新分配 | ✅ 自动适应核心数 |
| **调试难度** | ✅ 容易 | ❌ 地狱级（偶发 Bug） | ✅ 任务独立容易定位 |
| **适用场景** | 简单工具 | 特定场景优化 | **游戏引擎首选** |

### 游戏引擎为什么一定要用 JobSystem？

一个现代游戏每帧要做的事：

```
一帧（16ms，60 帧）：
  物理模拟：几百个物体的碰撞检测
  AI 更新：几十个敌人的决策逻辑
  动画更新：角色骨骼动画计算
  粒子系统：成千上万的粒子更新
  网络同步：发送和接收数据
  场景剔除：判断哪些物体可见
  ……
  
如果单线程：16ms 根本不够用
如果手动多线程：程序员疯掉
用 JobSystem：把所有任务扔进去，自动分配
```

## 七、一句话总结

> **JobSystem 就像一个聪明的包工头——你只需要把活拆成一个个小任务交给他，他会自动分配给所有空闲的工人（CPU 核心）。工人干完了自己的活，还会主动去帮别人（工作窃取）。结果是：没有一个工人摸鱼，CPU 所有核心都在全速运转。**
>
> **下次打开任务管理器看到所有核心都在 90% 以上——别慌，这可能是 JobSystem 在高效工作，而不是电脑中毒了。** 🚀
