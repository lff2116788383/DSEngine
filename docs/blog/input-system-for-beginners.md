## 一、从物理按键到游戏动作：键盘的 W → 角色"向前走"

### 四层转换

```
你按下 W 键 → 角色向前走：

物理层（Physical Layer）：
  你按下 W 键
  → 键盘电路接通
  → 键盘向电脑发送一个"数据包"
  → 数据包里写着："按键 0x11（W 键）被按下"

系统层（System Layer）：
  操作系统（Windows/macOS/Linux）收到按键事件
  → 把事件传给当前正在运行的游戏窗口
  → 游戏收到："按键 W 被按下"

引擎层（Engine Layer）：
  游戏引擎（的输入系统）收到按键事件
  → 查询"按键映射表"
  → W 键 = "向前移动"动作
  
游戏逻辑层（Game Logic Layer）：
  角色的移动系统收到"向前移动"指令
  → 根据当前速度和方向
  → 更新角色位置
  → 角色向前走了一小步
```

### 每一层在做什么

```
你按 W：
  │
  ▼
[键盘]          → 物理层：按键被按下，生成电信号
  │
  ▼
[驱动程序]      → 驱动层：把电信号翻译成"按键码"
  │
  ▼
[操作系统]      → 系统层：把按键码传给游戏窗口
  │
  ▼
[游戏引擎输入系统] → 引擎层：把按键码翻译成"动作"
  │
  ▼
[游戏逻辑]      → 逻辑层：执行"向前移动"行为
  │
  ▼
角色向前走 🚶
```

### 用生活场景理解

📞 **打电话：**

```
你拿起电话拨号：

  你拨了号码 10086（物理按键）
  → 电话把你的按键变成电信号（物理层）
  → 电信系统把信号路由到 10086（系统层）
  → 10086 客服接听（引擎层）
  → 客服处理你的请求（逻辑层）

每一层各司其职
你不需要知道电信系统怎么工作的
你只需要按号码
```

## 二、Action Mapping：把"按 W"映射为"向前移动"

### 直接的问题

如果没有 Action Mapping，游戏代码会写成这样：

```cpp
// 最原始的方式：直接判断按键
function handleInput(key):
    if key == KEY_W:
        player.moveForward()
    elif key == KEY_A:
        player.moveLeft()
    elif key == KEY_S:
        player.moveBackward()
    elif key == KEY_D:
        player.moveRight()
```

问题来了——**如果玩家想用 WASD，但另一部分玩家想用方向键呢？**

```cpp
// 支持两种按键方案
function handleInput(key):
    if key == KEY_W or key == KEY_UP:
        player.moveForward()
    elif key == KEY_A or key == KEY_LEFT:
        player.moveLeft()
    // ……
    
    // 如果玩家想自定义按键呢？
    // 那就得写一个配置文件，把这堆判断改成查表
```

### Action Mapping 的思路

> **不直接说"按 W 键"，而是说"执行向前移动这个动作"。至于哪个键对应这个动作——可以随便配。**

```
Action Mapping 的工作方式：

  定义动作（Action）：
    "向前移动" = 一个抽象的动作名称
    
  绑定按键（Binding）：
    默认：W 键 → "向前移动"
    也可以：上方向键 → "向前移动"
    也可以：手柄左摇杆上推 → "向前移动"
    也可以：你自己随便配

  游戏逻辑只关心"动作"：
    if 动作"向前移动"被触发:
        player.moveForward()
    
  游戏逻辑不需要知道玩家按了哪个键！
```

### Action Mapping 的配置

```cpp
// Action Mapping 的配置文件（可以被玩家修改）
action_mappings = {
    // 移动
    "MoveForward":  { keys: ["W", "UP"],           gamepad: "leftStickUp" },
    "MoveBackward": { keys: ["S", "DOWN"],          gamepad: "leftStickDown" },
    "MoveLeft":     { keys: ["A", "LEFT"],          gamepad: "leftStickLeft" },
    "MoveRight":    { keys: ["D", "RIGHT"],         gamepad: "leftStickRight" },
    
    // 动作
    "Jump":         { keys: ["SPACE"],              gamepad: "A" },
    "Attack":       { keys: ["MOUSE_LEFT"],         gamepad: "RT" },
    "Reload":       { keys: ["R"],                  gamepad: "X" },
    "Interact":     { keys: ["E", "F"],             gamepad: "Y" },
    "Pause":        { keys: ["ESC", "P"],           gamepad: "START" },
}
```

### Action Mapping 的好处

```
① 支持任意输入设备
   
   玩家 A：用键盘鼠标 → W 键向前
   玩家 B：用手柄 → 左摇杆向前
   玩家 C：用方向盘 → 油门踏板向前
   
   游戏逻辑：都处理"向前移动"动作
   不需要为每种设备写不同的逻辑

② 玩家可以自定义按键
   
   玩家觉得 W 太远 → 改成 E 键向前
   只需要改映射表
   不需要改游戏代码

③ 同一个动作可以多个按键
   
   按 W 或按上方向键都能向前走
   在 Action Mapping 里配一下就行
   不需要写 if-else
```

## 三、输入录制与回放：自动测试和过场动画的基石

### 什么是输入录制？

> **把玩家一段时间内的所有输入操作记录下来，以后可以"重放"这些操作。**

```
输入录制的过程：

  第 1 帧：玩家按下 W
  第 2 帧：玩家仍然按着 W
  第 3 帧：玩家按下了 SPACE（跳跃）
  第 4 帧：玩家松开了 W
  ……
  
  录制结果：
    frame 1:  [W pressed]
    frame 2:  [W held]
    frame 3:  [W held, SPACE pressed]
    frame 4:  [W released, SPACE held]
    frame 5:  [SPACE held]
    frame 6:  [SPACE released]
    ……
```

### 输入录制有什么用？

```
① 自动测试（最重要）
   
   让游戏自动跑一遍"录制好的操作"
   检查每一步是否正确
   
   比如：
     录制"从 A 点跑到 B 点，捡起道具，打开菜单"
     每次编译后重放这个录制
     检查是否还能正常完成
     → 自动化回归测试 ✅

② 过场动画中的"玩家操作"
   
   有些过场动画里，角色跟着玩家操作走：
     需要玩家的按键动作触发过场
     但过场又是预设的
     → 可以用录制的输入来驱动过场动画

③ 回放系统（Replay）
   
   《超级马里奥赛车》的回放：
     录下玩家每一帧的操作
     回放时"重放"这些操作
     就能完美复现整场比赛
     
   《守望先锋》的精彩回放也是同理
```

### 伪代码

```cpp
// 输入录制
function recordFrame():
    record = {
        frameNumber: currentFrame,
        keysDown: getAllPressedKeys(),
        mousePosition: getMousePosition(),
        mouseButtons: getPressedMouseButtons(),
        gamepadState: getGamepadState(),
    }
    recording.append(record)

// 输入回放
function playbackFrame():
    record = recording[currentFrame]
    for each key in record.keysDown:
        simulateKeyPress(key)
    setMousePosition(record.mousePosition)
    // ……模拟所有输入
    // 游戏会把这段回放当成"真实输入"处理
```

## 四、输入缓冲：格斗游戏里的"提前输入"是怎么回事

### 格斗游戏的特殊需求

玩过《拳皇》或《街霸》吗？你有没有遇到过这种情况：

> **你想搓一个"必杀技"（比如 →↓↘ + 拳），但明明搓对了，角色却没放出来。**

原因很可能是**输入时机不对**——你的操作比游戏判定早了那么几帧。

### 输入缓冲（Input Buffer）的解决方案

> **把玩家的输入"存"一下，不是马上执行，而是保留一小段时间。**

```
没有输入缓冲：

  第 1 帧：玩家按下"→"
  第 2 帧：玩家按下"↓"  → 游戏判定：→↓ 不完整，什么也不做
  第 3 帧：玩家按下"↘"
  第 4 帧：玩家按下"拳"
  
  → 必杀技没放出来！因为→↓↘这个序列被帧间隔打断了

有输入缓冲（3 帧缓冲）：

  第 1 帧：玩家按下"→"   → 缓冲：[→]
  第 2 帧：玩家按下"↓"   → 缓冲：[→, ↓]
  第 3 帧：玩家按下"↘"   → 缓冲：[→, ↓, ↘]
  第 4 帧：玩家按下"拳"   → 缓冲：[→, ↓, ↘, 拳]
                          → 检查缓冲序列：→↓↘+拳 = 必杀技！
                          → 必杀技释放成功！✅
```

### 输入缓冲的时间

```
不同的游戏类型，输入缓冲的时长不同：

  格斗游戏：3~5 帧（约 50~80ms）
    → 让搓招更"宽容"，不容易搓不出来
    → 但太长了会感觉"操作不跟手"
  
  FPS 射击游戏：几乎没有缓冲（0~1 帧）
    → 开枪必须秒响应
    → 有缓冲会感觉"开枪慢了"
  
  动作游戏（《只狼》《黑魂》）：2~3 帧
    → 格挡和翻滚需要"预输入"
    → 但又不能太宽容，否则失去挑战性
```

### 用生活场景理解

🎮 **格斗游戏的输入缓冲就像"快捷键"：**

```
你在手机上打字：

  没有输入缓冲：
    按一个键 → 立刻显示一个字母
    慢一点都打错
  
  有输入缓冲（就像手机的"预测输入"）：
    你快速打了 "helo"
    手机：你是不是想打 "hello"？
    自动纠正成 "hello"
    
  输入缓冲就是格斗游戏的"自动纠错"
  让你搓招时不用那么精确
```

## 五、生活中的类比：翻译官、遥控器、录音机

### 类比 1：翻译官 🌐

```
输入系统 = 翻译官

  键盘（说英语） → 翻译官（输入系统） → 游戏逻辑（说中文）
  
  你按 W（英语的 "W"）
  翻译官（Action Mapping）告诉游戏逻辑：
    "他说的是'向前走'"
    
  游戏逻辑（只懂中文）收到"向前走"
  → 执行向前移动

换一个键盘（法语键盘）：
  同样的位置但按键不同
  翻译官（Action Mapping）重新映射一下
  游戏逻辑不需要任何改变
```

### 类比 2：万能遥控器 📺

```
万能遥控器 = Action Mapping

  空调：按"模式"键 → 切换制冷/制热
  电视：按"模式"键 → 切换输入源
  音响：按"模式"键 → 切换音效模式
  
  同一个"模式"按键，在不同设备上有不同的含义
  这和 Action Mapping 完全一样：
  
  同一个"Space"键：
    在游戏中 → 跳跃
    在打字中 → 空格
    在浏览器中 → 滚动页面
```

### 类比 3：录音机 🎙️

```
录音机 = 输入录制/回放

  录音（录制）：
    把你的声音保存下来
    = 把玩家的输入保存下来
  
  播放（回放）：
    把保存的声音播放出来
    = 把保存的输入重新发送给游戏
  
  录音可以反复听
  录制好的输入可以反复测试
```

## 六、对比表格：轮询 vs 事件驱动

| 对比维度 | 轮询（Polling） | 事件驱动（Event-Driven） |
|:--------|:--------------|:----------------------|
| **工作方式** | 每帧检查"所有按键的状态" | 只有按键状态变化时才通知 |
| **性能** | ❌ 每帧都在查，有浪费 | ✅ 有变化才处理 |
| **实现难度** | ✅ 简单 | ❌ 需要事件队列 |
| **是否丢帧** | ❌ 不会丢（每一帧都查了） | ⚠️ 如果事件队列满了可能丢 |
| **适合场景** | **游戏主循环（每帧都要处理输入）** | UI 交互、窗口事件 |

### 游戏里两种都用

```
实际上游戏输入系统是"轮询 + 事件驱动"混用的：

轮询部分：
  每帧检查键盘上的所有按键
  "W 键是不是按着的？"
  "A 键是不是按着的？"
  
  因为游戏每帧都要知道"当前的输入状态"
  轮询是最可靠的方式

事件驱动部分：
  当按键被按下/释放时
  触发一个事件
  不需要每帧去查"按键是不是刚被按下"
  
  处理"按键变化"的问题
```

## 七、一句话总结

> **输入系统就是游戏世界的"翻译官"——它把键盘的 W 键、手柄的摇杆、鼠标的移动，统一翻译成游戏能理解的"向前走""转身""开火"等动作。Action Mapping 让玩家可以自定义按键，输入录制让游戏可以"倒带"重放操作，输入缓冲让格斗游戏的搓招不再反人类。没有输入系统，你按的每一个键都要游戏程序员写一堆 if-else 来处理。**
>
> **下次你在游戏里自定义按键布局时——你正在修改的不是游戏逻辑，而是输入系统的"翻译词典"。** 🎮
