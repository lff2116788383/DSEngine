# LOD——远处的山为什么看不清？

> **灵魂拷问：你站在山顶眺望远方，远处的树和近处的树——是同一棵吗？**

> 当然不是。但你有没有想过：**游戏里为什么远处的东西看起来"模糊"？** 不是因为你的眼睛有问题，而是因为游戏在"偷懒"——它在用**更简单的模型**来画远处的东西，节省性能的同时你还看不出来。

## 一、近处高清、远处低清——性能的"精打细算"

### 游戏性能的残酷现实

一个游戏场景里可能会有：

```
一片森林：
  近处的树：10 棵
  中距离的树：50 棵
  远处的树：500 棵
  最远处的树：5000 棵

总共：5560 棵树
```

如果每棵树都用最高精度的模型（5 万个三角形）——

```
每棵树 × 5 万三角形 = 2.78 亿个三角形

你的显卡：😱
```

### 但关键是——远处的树你根本看不清细节

```
你站在这里：
  ● ← 你
  
  距离 10 米：这棵树有 50 万像素
  你可以看清每一片叶子 🌿  → 需要高精模型

  距离 100 米：这棵树只占 5000 像素
  你能看到树的轮廓 🌳  → 中精模型就够了

  距离 500 米：这棵树只占 100 像素
  你只能看到一个绿点 🌲  → 低精模型甚至纸片都行
```

### LOD 的哲学

> **不要在不值得的地方浪费性能。**

LOD（Levels of Detail，细节级别）的核心思路就是：

> **近处的物体用高精度模型，远处的物体用低精度模型——反正你看不出来。**

## 二、LOD 的核心原理：模型越远越简单

### LOD 的工作方式

```
LOD 0（最近，最高精度）
  ┌──────────────────────────────┐
  │  三角形数量：10,000 个         │
  │  可以看到：树叶的纹理、树枝    │
  │  切换距离：0 ~ 20 米          │
  │  占屏幕面积：大               │
  └──────────────────────────────┘

LOD 1（中距离，中等精度）
  ┌──────────────────────────────┐
  │  三角形数量：2,000 个          │
  │  可以看到：树的轮廓，大致形状  │
  │  切换距离：20 ~ 60 米         │
  │  占屏幕面积：中               │
  └──────────────────────────────┘

LOD 2（远距离，低精度）
  ┌──────────────────────────────┐
  │  三角形数量：300 个            │
  │  可以看到：一个树形            │
  │  切换距离：60 ~ 150 米        │
  │  占屏幕面积：小               │
  └──────────────────────────────┘

LOD 3（最远，极低精度）
  ┌──────────────────────────────┐
  │  三角形数量：2 个（一个纸片）   │
  │  可以看到：一个绿色的三角形    │
  │  切换距离：150 米以上          │
  │  占屏幕面积：几乎看不见        │
  └──────────────────────────────┘
```

### 三级 LOD 的视觉对比

      ╱    ╲

### LOD 怎么切换？

```cpp
// LOD 切换逻辑（伪代码）
function getLODLevel(object, cameraPosition):
    // 计算物体离相机有多远
    distance = length(object.position - cameraPosition)
    
    // 根据距离选择 LOD 级别
    if distance < 20:
        return LOD_0  // 最高精度
    elif distance < 60:
        return LOD_1  // 中等
    elif distance < 150:
        return LOD_2  // 低精度
    else:
        return LOD_3  // 纸片
```

### 性能节省有多大？

```
一棵树：
  LOD 0：10,000 三角形
  LOD 1：2,000 三角形
  LOD 2：300 三角形
  LOD 3：2 三角形（纸片）

一片森林（5000 棵树）：
  
  无 LOD（全部用 LOD 0）：
    5000 × 10,000 = 5000 万三角形 ❌
  
  有 LOD：
    近处 10 棵 × LOD 0 = 10 万
    中距 50 棵 × LOD 1 = 10 万
    远处 500 棵 × LOD 2 = 15 万
    最远 4440 棵 × LOD 3 = 8880
    合计 ≈ 36 万三角形 ✅
    
  节省了 99.3% 的性能！
```

## 三、Hysteresis（迟滞）：防抖动——防止来回切换

### 一个潜在的问题

想象你在游戏里向前走：

```
你向前走 1 米：
  一棵树在 19.5 米 → LOD 0 ✅
  你再往前 0.6 米 →
  树变成了 20.1 米 → LOD 1 ← 切换了
  
  但如果你后退 0.1 米 →
  树变回 20.0 米 → LOD 0 ← 又切换回来了！
  再往前 0.1 米 →
  树变回 20.1 米 → LOD 1 ← 又切换了！
```

**这就是"抖动"问题：** 物体在 LOD 边界上来回切换，画面会出现"闪烁"——一会儿高清一会儿低清，非常影响体验。

### Hysteresis 的解决方案

**Hysteresis（迟滞/滞后）** 的原理很简单：

> **切换点不一样——从低到高和从高到低用不同的距离阈值。**

```
没有 Hysteresis：
          LOD 0            LOD 1
  ─────────●───────────────●─────────→
         20 米            60 米
         
  物体在 19.9 米 → LOD 0
  往前走 0.2 米 → 20.1 米 → LOD 1
  后退 0.2 米 → 19.9 米 → LOD 0 (又切回来了)

有 Hysteresis：
          LOD 0            LOD 1
  ─────────●───────●───────●─────────→
         18 米  20 米    60 米
         ↑切换点 ↑原始   ↑切换点
         (LOD1→0) 边界   (LOD0→1)

  从近到远：
    18 米以内 → 一定 LOD 0
    20 米开始 → 切换到 LOD 1
  
  从远到近：
    20 米 → 仍然是 LOD 1（不会切换）
    只有到 18 米 → 才切换回 LOD 0
  
  结果：
    如果你在 19 米附近来回走
    → 不会来回切换 ✅
```

### 用生活场景理解

🌡️ **空调的温度控制：**

```
你设了空调 26°C：

没有 Hysteresis：
  26.1°C → 空调制冷
  25.9°C → 空调停止
  26.1°C → 空调又开始制冷
  25.9°C → 又停止
  → 空调开关开关开关…… 💥

有 Hysteresis：
  27°C → 空调开始制冷
  25°C → 空调停止
  → 中间有 2°C 的缓冲区
  → 空调不会频繁开关 ✅
```

LOD 的 Hysteresis 也是一样的道理：**在切换点附近留一个缓冲区，防止频繁切换。**

## 四、屏幕空间投影公式的直观理解

### 不只是看距离

你可能会觉得：**LOD 就是看距离嘛，远了就用低模。**

但实际情况更精细一些——LOD 不只看**物体离相机多远**，还要看**物体在屏幕上占多大**。

### 为什么只算距离不够？

```
情况 1：一个巨大的建筑在 100 米外
  它仍然占了屏幕的 30%
  → 需要用高 LOD（LOD 0）

情况 2：一只蚂蚁在 1 米外
  它只占了屏幕的 0.1%
  → 用低 LOD 完全够（LOD 3）

如果只用"距离"判断：
  建筑 100 米 → 判定为"远" → 用低模 → 画面里建筑模糊了 ❌
  蚂蚁 1 米 → 判定为"近" → 用高模 → 浪费性能 ❌
```

### 屏幕空间投影

正确的做法是：**算物体在屏幕上实际占了多少像素。**

```cpp
// 直观理解（不是真实公式）
function getScreenSize(object, camera):
    // 物体离相机的距离
    distance = length(object.position - camera.position)
    
    // 物体的实际大小
    realSize = object.boundingSphere.radius
    
    // 屏幕上的大小 ≈ 物体大小 ÷ 距离
    screenSize = realSize / distance
    
    // 再乘以屏幕分辨率等参数
    pixelCount = screenSize * screenWidth * someFactor
    
    return pixelCount
```

公式的直观理解：

> **屏幕上的大小 ≈ 物体的真实大小 ÷ 离相机的距离 × 屏幕参数**

```
一个 10 米大的建筑：
  在 10 米外 → 10 / 10 = 1.0 → 占了屏幕很大面积 → LOD 0
  在 100 米外 → 10 / 100 = 0.1 → 占了屏幕小面积 → LOD 2
  在 1000 米外 → 10 / 1000 = 0.01 → 几乎看不见 → LOD 3

一只 0.1 米大的蚂蚁：
  在 1 米外 → 0.1 / 1 = 0.1 → 很小 → LOD 2
  在 10 米外 → 0.1 / 10 = 0.01 → 看不见了 → LOD 3
```

### LOD 选择的总原则

```
不是"离得远就用低模"
而是"在屏幕上看不清细节的地方就用低模"
```

| 因素 | 对 LOD 的影响 |
|:----|:------------|
| 📏 **物体大小** | 越大 → 越需要高 LOD |
| 📐 **距离** | 越远 → 可以用低 LOD |
| 🖥️ **屏幕分辨率** | 越高 → 需要更精细的 LOD |
| 👁️ **视野中心** | 中心区域 → 高 LOD；边缘 → 低 LOD |
| 🎬 **运动速度** | 运动越快 → 可以用低 LOD（看不清细节） |

## 五、生活中的类比：近视眼、手机相册

### 类比 1：近视眼 👓

```
不戴眼镜看远处：
  10 米内的人 → 能看清五官（LOD 0）
  20 米外的人 → 只能看清轮廓（LOD 1）
  50 米外的人 → 只能看到一个人形（LOD 2）
  100 米外的人 → 一个小点（LOD 3）

戴上眼镜：
  所有距离都能看清
  → 但你的"视野系统"（显卡）累坏了
```

**LOD 就是主动让自己"近视"**——远处的东西不需要看清。

### 类比 2：手机相册 📱

```
你手机里有一张 4000 万像素的照片：

📱 缩略图模式：
  每张照片只显示 100×100 像素
  → 但你看到的"内容"和原图一样
  → 省内存、加载快
  → 这就是 LOD 1/2

🖼️ 点击查看大图：
  显示 2000×2000 像素
  → 你能看到更多细节
  → 更耗内存
  → 这就是 LOD 0

"先显示缩略图，需要时再加载高清图"
= "远处用低模，近处再用高模"
```

### 类比 3：画家的技巧 🎨

```
画家画风景的时候：
  近处的树：
    画得很细致——树枝、树叶、纹理
  
  远处的山：
    用一块颜色带过——不需要细节
  
  这不是画家偷懒
  而是人类视觉系统本来就如此
```

**LOD 做的就是画家做的事——在远处省掉细节，反正你看不出来。**

## 六、对比表格：有 LOD vs 无 LOD

| 对比维度 | 无 LOD | 有 LOD |
|:--------|:------|:-------|
| **远处物体精度** | 和近处一样精细 | 简化模型或纸片 |
| **三角形数量** | 全部 × 最高精度 | 近处高精 + 远处低精 |
| **性能** | ❌ 场景一大就崩溃 | ✅ 可以渲染超大规模场景 |
| **画质损失** | ✅ 完美 | ⚠️ 几乎看不出（换 LOD 时可能有一瞬间变化） |
| **实现复杂度** | ✅ 不用管 | ❌ 需要多套模型 + 切换逻辑 |

### 无 LOD 的后果

```
《赛博朋克 2077》的一个城市街区：
  建筑：200 栋
  车辆：50 辆
  行人：100 个
  路灯、招牌、垃圾桶：500 个
  合计物体：约 850 个

如果每个物体都用最高精度：
  平均每个物体：50,000 三角形
  总计：850 × 50,000 = 4250 万三角形
  加上其他特效：直接爆显存 💥

用 LOD：
  近处：50 个物体 × 高精 = 250 万
  中距：200 个物体 × 中精 = 400 万
  远处：600 个物体 × 低精 = 180 万
  总计：830 万三角形 ✅ 流畅运行
```

### LOD 的未来

LOD 技术也在进化：

| 技术 | 说明 | 代表游戏/引擎 |
|:----|:-----|:------------|
| 🏛️ **传统 LOD** | 美术准备多套模型 | 几乎所有游戏 |
| 🔄 **自动 LOD** | 工具自动减面 | Unreal Engine 的 Simplygon |
| 🧩 **Nanite（虚拟微多边形）** | 自动按像素精度加载 | Unreal Engine 5 |
| 📦 **网格着色器** | GPU 实时决定细节 | 最新游戏 |

尤其是 Unreal Engine 5 的 **Nanite** 技术——它让 LOD 的切换完全自动化，美术不需要手动准备多套模型。但核心思想还是一样的：**近处用高精，远处用低精。**
