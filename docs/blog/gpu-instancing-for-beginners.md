# GPU Instancing——一千棵树一次画完

> **灵魂拷问：游戏里有一片森林，里面有 1000 棵树。怎么画最快？**
>
> 最笨的办法：一棵一棵画——CPU 喊 1000 次"画树！"，GPU 接 1000 次指令。每次喊话都有开销，1000 次下来光"喊话"就累死了。
>
> 聪明的办法：**GPU Instancing**——CPU 只喊一次"画树！这里有 1000 棵的数据"，GPU 一口气画完 1000 棵。

## 一、Draw Call 是什么？—— CPU 对 GPU 的"喊话"

### 什么是 Draw Call？

**Draw Call（绘制调用）** 就是 CPU 告诉 GPU "给我画这个东西"的指令。

```
CPU 对 GPU 喊话：
  "GPU，给我画一棵树！"
  GPU：收到！开画……画完了
  CPU："再画一棵树！"
  GPU：收到……又画完了
  CPU："再画一棵！"
  GPU：……
```

### 画一个物体需要的步骤

```
要画一棵树：

Step 1：CPU 准备数据
  ├─ 树的顶点位置在显存里
  ├─ 树的纹理贴图在显存里
  └─ 树的位置在 (x, y, z)

Step 2：CPU 发出 Draw Call
  ├─ 绑定顶点缓冲 → "顶点在这"
  ├─ 绑定纹理 → "贴图在这"
  ├─ 设置渲染状态 → "不开透明度"
  └─ Draw() → "画！"

Step 3：GPU 执行
  └─ 收到指令 → 开始渲染 → 输出像素
```

### Draw Call 的频率

```
一个简单的场景：
  地面：1 个 Draw Call
  建筑：10 个 Draw Call
  角色：5 个 Draw Call
  树木：50 个 Draw Call
  路灯：20 个 Draw Call
  粒子：1 个 Draw Call
  
  合计：87 个 Draw Call

一个复杂的场景（AAA 游戏）：
  场景中各种物体 → 2000~5000 个 Draw Call
```

## 二、为什么 Draw Call 多了会卡？

### 每个 Draw Call 都有"固定成本"

CPU 发出一个 Draw Call 不是免费的——它需要做很多准备工作：

```
CPU 发起一次 Draw Call 的开销：
  ┌──────────────────────────────────┐
  │  1. 验证参数（检查数据是否正确）   │
  │  2. 切换渲染状态（如果和上次不同）  │
  │  3. 绑定缓冲区（告诉 GPU 数据在哪）│
  │  4. 提交命令到 GPU 命令队列        │
  │  5. 上下文切换（可能涉及驱动）     │
  └──────────────────────────────────┘
  每次大概：0.01~0.1 毫秒
```

看起来很少对吧？但乘上几千次：

```
100 个 Draw Call：
  100 × 0.05ms = 5ms → 还行，在 16ms 预算内

500 个 Draw Call：
  500 × 0.05ms = 25ms → 超了！60fps 需要 16ms 以内

2000 个 Draw Call：
  2000 × 0.05ms = 100ms → 卡成幻灯片
```

### 用生活场景理解

📋 **点菜：**

```
普通渲染（没有 Instancing）：

你请 1000 个人吃饭：
  你走到第 1 个人面前： "你要吃什么？"
  第 1 个人： "鱼香肉丝"
  你去后厨下单
  
  你走到第 2 个人面前： "你要吃什么？"
  第 2 个人： "鱼香肉丝"
  你去后厨下单
  
  ……重复 1000 次……
  
  时间都花在"走"和"下单"上了，
  而不是花在做菜上！

这就是 Draw Call 太多的问题：
  时间都花在"CPU 给 GPU 发指令"上
  而不是花在"GPU 真正渲染"上
```

### 性能瓶颈在哪里？

```
Draw Call 太多时的性能瓶颈：
  不在 GPU（显卡）
  而在 CPU（处理器）！
  
CPU 忙死了：
  一直在发 Draw Call 指令
  没空做其他事（物理、AI、逻辑）

GPU 闲死了：
  每一次都瞬间画完
  然后等 CPU 发下一条指令
```

> **GPU Instancing 就是来解决这个问题的：让 CPU 只喊一次，GPU 画一千次。**

## 三、Instancing 怎么合并 Draw Call？

### Instancing 的思路

**GPU Instancing（实例化）** 的核心思想：

> **如果 1000 棵树用的是同一个模型、同一张贴图，为什么不让 GPU 一次画完？**

```
普通渲染：
  CPU → "画树" → GPU 画 1 棵
  CPU → "画树" → GPU 画 1 棵
  CPU → "画树" → GPU 画 1 棵
  ... 重复 1000 次 ...
  
  总开销：1000 次 Draw Call

Instancing：
  CPU → "画树，这里有 1000 个位置，一次画完"
  GPU → 一次性画完 1000 棵
  
  总开销：1 次 Draw Call
```

### Instancing 的执行流程

```
Step 1：CPU 准备数据
  ┌──────────────────────────────┐
  │  模型数据（只需一份）：        │
  │  ├─ 树的顶点数据             │
  │  ├─ 树的纹理贴图             │
  │  └─ 树的材质参数             │
  │                              │
  │  实例数据（每个实例一份）：    │
  │  实例 1：位置 (10,0,20)，旋转 30°│
  │  实例 2：位置 (15,0,25)，旋转 45°│
  │  实例 3：位置 (20,0,18)，旋转 60°│
  │  ... 共 1000 个实例           │
  └──────────────────────────────┘

Step 2：CPU 发出 1 次 Draw Call
  ┌──────────────────────────────┐
  │  DrawInstanced(1000)         │
  │  → 模型指针 → 顶点缓冲       │
  │  → 实例指针 → 变换数据      │
  │  → "一次画 1000 个"          │
  └──────────────────────────────┘

Step 3：GPU 执行
  ┌──────────────────────────────┐
  │  for i = 0 to 999:           │
  │    shader中使用实例 i 的变换  │
  │    渲染模型                   │
  │  → 1000 棵树一次画完          │
  └──────────────────────────────┘
```

### 性能对比

```
一片森林，1000 棵树，每棵树 1 万个三角形：

普通渲染：
  Draw Call：1000 次
  CPU 时间：1000 × 0.05ms = 50ms
  GPU 时间：渲染性能足够 → 瞬间画完
  瓶颈：CPU 被 Draw Call 拖死 ❌

Instancing：
  Draw Call：1 次
  CPU 时间：1 × 0.05ms = 0.05ms
  GPU 时间：几乎不变
  瓶颈：CPU 几乎零开销 ✅
```

## 四、Instancing 的原理：一份模型 + 一堆变换矩阵

### Instancing 的核心数据

Instancing 需要两种数据：

```
① 共享数据（所有实例相同）：
   ┌────────────────────┐
   │ 顶点位置（模型本身）│
   │ 法线数据          │
   │ UV 坐标           │
   │ 纹理贴图          │
   │ 材质参数          │
   └────────────────────┘

② 实例数据（每个实例不同）：
   ┌────────────────────┐
   │ 实例 1 的变换矩阵   │ → 位置、旋转、缩放
   │ 实例 2 的变换矩阵   │ → 位置、旋转、缩放
   │ 实例 3 的变换矩阵   │ → 位置、旋转、缩放
   │ ...                │
   └────────────────────┘
```

### 变换矩阵是什么？

每个实例需要知道"自己在哪里、多大、朝哪"——这个信息用一个 **4×4 的变换矩阵** 表示。

```
一个变换矩阵包含：
  ┌──────────────────────┐
  │  1.0  0.0  0.0  10.0 │  ← 旋转 + 缩放（3×3）
  │  0.0  0.5  0.0  0.0  │     位置（最后一列）
  │  0.0  0.0  1.0  20.0 │
  │  0.0  0.0  0.0  1.0  │
  └──────────────────────┘
  → 这个实例位于 (10, 0, 20)，Y 方向压扁了一半
```

### CPU 到 GPU 的数据传输

```
CPU 准备：
  1000 个变换矩阵
  = 1000 × 16 个 float
  = 1000 × 64 字节
  = 64 KB
  
  64 KB → 一次性传给 GPU ✅ 很快

普通渲染 1000 次 Draw Call 的数据传输：
  每次要传模型指针、纹理指针、状态……
  每次都要和 GPU "握手"
  1000 次握手 → 非常慢 ❌
```

### Instancing 的适配条件

```
Instancing 最适合的场景：
  ✅ 大量相同的模型——森林、草地、石头、路灯
  ✅ 每个实例只有位置/旋转/缩放不同
  ✅ 使用相同的材质和纹理
  ✅ 不需要单独控制每个实例的动画

Instancing 不太适合的场景：
  ❌ 每个物体完全不同（一个树、一个房子、一个角色）
  ❌ 每个物体有不同的材质和纹理
  ❌ 需要单独控制每个物体的骨骼动画
```

### 实例化的变种

| 技术 | 说明 | 适用场景 |
|:----|:-----|:--------|
| **Basic Instancing** | 相同模型 + 不同位置 | 森林、石头 |
| **Indirect Instancing** | GPU 自己决定画什么 | 粒子系统、草丛 |
| **Mesh Instancing** | 多个不同模型合并 | 复杂场景 |
| **Texture Array Instancing** | 不同实例用不同贴图 | 多种树木混种 |

## 五、生活中的类比：印刷 vs 手写、集体照

### 类比 1：印刷 vs 手写 📄

```
你要发 1000 份通知：

手写（普通渲染）：
  第 1 份：拿起笔，从头写到尾，放好
  第 2 份：拿起笔，从头写到尾，放好
  ……重复 1000 次……
  时间主要花在"拿起笔"和"放好"上

印刷（Instancing）：
  设计好模板 → 一次印刷 1000 份
  
  模板 = 树的模型
  印刷机的参数调校 = 每个实例的位置
  一次印刷 = 一次 Draw Call
```

### 类比 2：集体照 📸

```
全班 50 个人拍合照：

一个一个拍（普通渲染）：
  拍第 1 个人 → 换人 → 拍第 2 个人 → 换人……
  拍完 50 张照片，每张都是不同的背景
  摄影师累死了

集体照（Instancing）：
  所有人站好 → 一次拍完
  → 一张照片里 50 个人，每个人位置不同
  → 摄影师只按了一次快门
  
  快门 = Draw Call
  每个人的位置 = 实例的变换矩阵
  同一次快门 = Instancing
```

### 类比 3：阅兵方阵 🪖

```
阅兵式上，一个方阵 1000 个士兵：

没有 Instancing：
  指挥官一个一个喊：
  "士兵 1，出列，正步走！"
  "士兵 2，出列，正步走！"
  ……嗓子喊哑了

Instancing 模式：
  指挥官喊一次：
  "方阵，全体都有，正步走！"
  → 所有士兵同时前进
```

## 六、对比表格：普通渲染 vs Instancing

| 对比维度 | 普通渲染 | GPU Instancing |
|:--------|:--------|:--------------|
| **Draw Call 数量** | 物体数量 × 1 | 1（或很少） |
| **CPU 开销** | 高（每个物体都要发指令） | 极低（一次准备全部数据） |
| **GPU 开销** | 相同 | 相同 |
| **适用条件** | 任何物体 | 相同模型 + 相同材质 |
| **数据传输** | 每个物体单独传 | 一次传所有实例数据 |
| **灵活性** | ✅ 每个物体可以完全不同 | ❌ 只能改位置/旋转/缩放 |
| **内存占用** | 低 | 略高（需要存实例数据） |
| **性能提升** | 基线 | 同样资源下可画 10~100 倍物体 |

### 实际游戏中的使用

```
《刺客信条》的城市场景：
  路灯：500 个
  → 同一个路灯模型
  → Instancing 渲染
  → 1 个 Draw Call ✅

《巫师 3》的森林：
  树木：3000 棵
  → 5 种树木模型
  → 每种模型 Instancing
  → 5 个 Draw Call ✅

《战地》的草地：
  草叶：数百万
  → 每片草是一个实例
  → Indirect Instancing
  → 几个 Draw Call 搞定 ✅
```

### 如果在不支持 Instancing 的年代

```
2005 年的游戏《魔兽世界》：
  没有 Instancing 技术
  
  一个森林场景：
    50 棵树 → 50 个 Draw Call
    不是 5000 棵树
    
  因为 CPU 扛不住 5000 个 Draw Call
  所以游戏里森林通常很稀疏

2020 年的游戏《地平线：零之曙光》：
  有 Instancing 技术
  
  一个森林场景：
    上万棵树 → 几十个 Draw Call
    茂密的森林成为可能
```

## 七、一句话总结

> **GPU Instancing 就是让 CPU 只喊一次"开画！"，然后 GPU 一口气画完一千个同样的物体。它把性能瓶颈从"CPU 忙着发指令"变成了"GPU 专心画画"，让你能用同样的硬件画出多出一百倍的树木、草地和石头。**
>
> **下次在游戏里看到一片茂密的森林——记住，那不是一棵一棵画出来的，而是 Instancing 一声令下，千万棵树同时"长"出来的。** 🌲
