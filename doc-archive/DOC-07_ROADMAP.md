# DOC-07 后续路线图

本文档基于当前仓库真实代码状态，给出面向**个人独立开发者**与**中小型工作室**的近中期路线。路线重点不是继续扩张功能面，而是把当前主线做成更稳定、更高效、更可信的开发闭环。

## 1. 当前阶段判断

当前项目处于以下阶段：

- 2D Runtime 主链已基本成型
- Lua / C++ 双业务宿主已经可用
- Editor 基础闭环已可用，但工程化程度不足
- 测试入口已建立，但门禁和持续集成仍需加强
- 资源系统已进入工程化阶段，但仍保留明显全局态
- 3D 已接入部分组件、模块、样例和测试入口，但不是默认稳定主线

因此，当前版本最准确的整体判断是：

**适合继续聚焦打磨，不适合继续无节制扩功能面。**

## 2. 路线总原则

后续路线坚持以下原则：

1. 以当前代码主线为准，不把远期目标写成已完成事实
2. 优先服务 2D 项目与轻量开发工作流
3. 优先提升可交付性、开发效率和稳定性
4. 3D 只推进到最小可验证 MVP，不做超前承诺
5. 文档、构建脚本、测试门禁保持统一口径

## 3. P1：2D 主线收口（最高优先级）

目标：让独立开发者与中小团队可以稳定使用 DSEngine 完成一个小型到中小体量 2D 项目闭环。

### 3.1 Runtime 与资源链路

重点包括：

- 继续推进实例化解耦
- 弱化 `AssetManager::Instance()` 等全局单例依赖，优先收口到 Runtime / Lua binding / System 注入边界
- 已完成 Lua ECS binding、SpineSystem、MeshRenderSystem、AnimatorSystem、Particle3DSystem、AudioSystem 的高价值资源访问收口
- 统一 data root 相对路径规范，调用层默认使用如 `models/cube.dmesh`、`script/main.lua` 的逻辑路径
- 已清理 sample 中 `data/...` 回退写法，并将音频 / 动画 / 粒子 / 网格主链切到统一逻辑路径解析模型
- 补齐更清晰的资源导入规则
- 为后续 meta / UUID / 依赖追踪预留结构

### 3.2 2D 模块稳定性

重点包括：

- 稳定 UI / Localization / Particle / Physics2D / Tilemap 的主链回归
- 补足多分辨率、遮罩、文本同步、粒子压力场景等高价值回归
- 统一样例与测试中的资源使用方式

### 3.3 最小交付口径

P1 完成后，应该能够明确对外说明：

- 2D Runtime 主链稳定可用
- Lua 玩法迭代可支撑日常开发
- 基础资源与测试路径可重复验证
- AssetManager 路径规范与关键系统注入边界已基本建立
- 小型 2D 项目具备可持续迭代基础

## 4. P2：Editor 工程化增强

目标：把当前“能用”的编辑器推进到“高频开发可用”的状态。

### 4.1 编辑器结构整理

重点包括：

- 拆分 `apps/editor_cpp/src/main.cpp`
- 拆出 editor_state、scene_io、hierarchy_panel、inspector_panel、viewport_panel、profiler_panel 等模块
- 降低单文件集中度与维护成本

### 4.2 基础编辑闭环增强

重点包括：

- 巩固实体创建、删除、复制、选中、重命名
- 强化 Transform 编辑体验
- 强化场景保存 / 加载稳定性
- 增加常见组件向导与默认模板

### 4.3 模块编辑器化

重点包括：

- UI Inspector 配置入口增强
- Particle 参数面板增强
- Localization 与 UI 文本绑定体验增强
- Profiler 历史曲线、导出与对比增强

### 4.4 暂不承诺的编辑器能力

当前阶段不建议对外承诺：

- 完整 Prefab 工作流
- 完整 Undo / Redo 指令体系
- 完整 Play In Editor 隔离
- 完整资源数据库与发布工具链

## 5. P3：测试、构建与性能门禁

目标：让项目从“有测试”升级为“有稳定门禁”。

### 5.1 测试门禁

重点包括：

- 把 `engine.unit`、`engine.lua_runtime`、`engine.spine` 和关键 2D smoke 作为基础回归门禁
- 区分日常快速回归与完整回归集合
- 固化 Debug / Release 双配置验证路径

### 5.2 构建脚本整理

重点包括：

- 优化 `build_all.bat` 的职责边界
- 强化失败摘要与错误可读性
- 保留针对本地开发的快速脚本入口
- 逐步让脚本与文档、CTest 口径一致

### 5.3 性能基线

重点包括：

- 为 UI、Particle、Lua 多实例等典型场景建立基础性能基线
- 继续增强 CPU / Memory / Render Profiler 的可观测性
- 逐步接入持续化性能回归检查

## 6. P4：3D MVP 收口（在前述阶段稳定后推进）

3D 当前状态应准确理解为：

- 已接入：组件层、部分渲染路径、动态模块、部分测试与样例
- 未收口：默认构建、资源主格式、编辑器闭环、验证门禁、统一工作流

因此 3D 的下一步不是继续扩功能面，而是做 **MVP 收口**。

### 6.1 3D 当前代码事实

当前可确认的事实包括：

- 存在 3D ECS 组件层
- 存在 `modules/gameplay_3d/` 模块代码
- `DSE_ENABLE_3D` 默认关闭
- 存在动态模块接口与按需加载方向
- 已有部分 3D 测试文件与样例入口
- 编辑器已接入部分 3D Inspector 与 Gizmo 交互基础

### 6.2 3D MVP 目标

只有达到以下条件，才应把 3D 从“已接入”升级为“最小可验证 MVP”：

- 可以通过明确脚本或构建开关打开 3D 验证流程
- 存在稳定的基础 3D 场景：Mesh + Camera + Light + Shadow + Skybox
- 存在最小 3D 场景保存 / 加载回归
- 存在至少一条稳定可跑的 3D smoke 链路
- 文档与代码状态一致，不再出现超前描述

### 6.3 当前不纳入 3D MVP 的方向

当前阶段不建议纳入 3D MVP：

- 完整 GI
- 完整 Shader Graph
- 大规模地形生态系统
- 全套后处理体系
- 完整 3D 商业级工具链

## 7. Launcher 路线

当前启动器主线为 `apps/launcher_tauri/`。

近程目标建议聚焦：

- 项目入口整合
- 版本信息展示
- 本地项目与模板管理
- 与本地引擎目录联动

不建议在当前阶段对外承诺：

- 账号系统
- 商城系统
- 增量更新平台
- 大规模在线服务能力

## 8. 文档路线

后续文档策略应继续收敛：

- README 只保留项目定位、边界、快速开始、核心文档入口
- 主线文档控制在少量核心文件内
- 模块能力作为专题文档按需维护
- 文档必须严格区分“已稳定 / 已实现 / 已接入 / 规划中”
- 不再把单独的 3D 路线图作为主线文档重点维护
- 新增版本规划文档 [`doc-archive/DOC-10_2D_AND_3D_MVP_VERSION_PLAN.md`](doc-archive/DOC-10_2D_AND_3D_MVP_VERSION_PLAN.md)，用于固定 2D 主线版本与 3D MVP 版本边界，避免路线无限扩张

## 9. 当前版本门禁建议

建议把以下项目作为阶段性门禁：

- `dse_engine` Debug 构建通过
- `dse_editor_cpp` Debug 构建通过
- `engine.unit` 通过
- `engine.lua_runtime` 通过
- `engine.spine` 通过
- 关键 2D smoke 通过
- 当前 README 与主线文档与代码状态一致

在 3D MVP 阶段，再增加：

- `engine.3d.*` 分组回归
- 3D 样例冒烟链路
- 3D 最小场景保存 / 加载回归

## 10. 总结

DSEngine 后续最重要的事情不是继续把功能列表写得更长，而是把以下几点做扎实：

- 先把 **2D 主线**做稳
- 再把 **Editor** 做顺手
- 再把 **测试、构建、性能门禁**做实
- 最后把 **3D** 从“已接入”推进到“最小可验证 MVP”

如果这条路线执行到位，DSEngine 会更符合它当前最真实、也最有竞争力的定位：

**一个面向个人独立开发者与中小型工作室的轻量级高性能 C++ 游戏引擎。**
