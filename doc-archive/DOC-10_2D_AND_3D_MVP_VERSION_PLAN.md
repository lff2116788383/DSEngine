# DOC-10 2D 主线与 3D MVP 版本规划

本文档用于把后续版本规划收敛到两个明确目标：

1. 先把 2D 主线做成可稳定交付版本
2. 再把 3D 推进到最小可验证 MVP 版本

本文档不继续无限制扩张功能列表，而是给出阶段边界、版本目标、验收口径与不纳入项。实际优先级以当前仓库代码状态为基础，并与 [`doc-archive/DOC-07_ROADMAP.md`](doc-archive/DOC-07_ROADMAP.md) 保持一致。

---

## 1. 当前版本判断

结合当前代码、测试、构建脚本与编辑器状态，项目更适合按以下节奏推进：

- `2D Stable` 作为当前最高优先级主线版本
- `Editor Usable` 作为服务 2D 主线的工程化增强版本
- `3D MVP` 作为在 2D 与基础工程能力稳定后的阶段性版本

这意味着：

- 现在不建议继续无限制增加新系统
- 现在应优先完成主线闭环、测试门禁、文档口径统一
- 3D 的目标不是“全面铺开”，而是“形成一个最小闭环版本”

---

## 2. 版本策略总原则

后续版本规划遵循以下原则：

1. 版本以“闭环能力”定义，而不是以“模块数量”定义
2. 2D 优先于 3D
3. 编辑器优先做高频闭环，而不是大而全工具链
4. 测试、构建、文档必须与版本口径同步
5. 每个版本都明确“不做什么”

---

## 3. 2D 主线版本规划

## 3.1 目标定位

2D 主线版本的目标不是展示尽可能多的功能，而是让一个小型到中小体量项目可以在当前引擎中完成：

- 场景编辑
- 运行时驱动
- Lua / C++ 业务迭代
- UI / 本地化 / 粒子 / 物理 / Tilemap 基础玩法
- 基本测试回归与构建验证

建议把 2D 主线拆为两个阶段版本。

### 3.1.1 V0.8 - 2D Stable Base

版本目标：形成“能稳定开发”的 2D 基础版本。

必须完成：

- Runtime 主循环稳定
- 2D 资源路径规范统一
- Lua Runtime 基础回归稳定
- UI / Localization / Particle / Physics2D / Tilemap 主链稳定
- 编辑器场景保存 / 加载 / 实体编辑主链稳定
- 关键测试目标可重复运行
- README 与主线文档口径一致

建议纳入门禁：

- [`dse_engine`](CMakeLists.txt)
- [`dse_editor_cpp`](apps/editor_cpp/CMakeLists.txt)
- `engine.unit`
- `engine.lua_runtime`
- 关键 2D smoke

不纳入该版本：

- 完整 Prefab 工作流
- 完整 Undo / Redo
- 完整资源数据库
- 完整 3D 工作流
- 商业级打包/发布系统

### 3.1.2 V0.9 - 2D Production Preview

版本目标：形成“适合持续制作”的 2D 预发布版本。

必须完成：

- 编辑器主入口进一步拆分，至少拆出 scene_io、toolbar、hierarchy、inspector、viewport 中的大部分高频模块
- 更稳定的 Play/Edit 体验
- 更清晰的构建与测试脚本职责边界
- 基础 profiler 指标可观测、可导出
- 样例、文档、构建方式统一

建议增加门禁：

- Debug / Release 双配置验证
- 更多 UI / Particle / Localization 回归
- 关键构建脚本失败摘要收口

不纳入该版本：

- 完整商业级编辑器生态
- 完整资源依赖数据库
- 大规模可视化脚本编辑器
- 全量高级 3D 能力

### 3.1.3 2D 正式可对外口径

当 V0.9 达到后，可以对外稳定表述：

- DSEngine 当前以 2D 为稳定主线
- 已具备 Lua / C++ 双宿主开发闭环
- 已具备基础编辑器工作流
- 已具备基础测试与构建门禁
- 适合独立开发者与中小团队做轻量 2D 项目

## 4. 3D MVP 版本规划

在进入版本定义前，先给出一个更贴近当前仓库状态的判断：

- **2D 主线：已基本完整，当前主要欠缺的是工程化收口，而不是主链功能模块本身**
- **3D 能力线：已接入基础组件、样例、测试与部分渲染链路，但距离“轻量级、高性能、现代化引擎”的 3D 目标仍有明显差距**

如果以“轻量级高性能现代化引擎”的视角衡量，当前 3D 仍主要缺少以下几类关键能力。

### 4.0 当前 3D 差距清单（面向轻量级高性能现代化引擎）

#### 4.0.1 渲染架构层
- 更清晰、可扩展的 Render Graph / Pass 组织边界
- 更稳定的材质系统与着色器变体管理
- 更系统的 PBR 渲染收口，而不是分散能力点
- 更完整的后处理链组织（而非仅具备局部能力）
- 更明确的渲染资源生命周期、缓存与复用策略
- 更强的 GPU 调试与渲染诊断能力

#### 4.0.2 性能能力层
- 更稳定可控的 3D 可见性裁剪链路（视锥裁剪、批次优化、实例化策略）
- 更明确的 CPU/GPU 性能基线与场景级 profiling 口径
- 更成熟的大场景资源加载与异步上传策略
- 更明确的性能门禁，而不仅是“能跑”
- 更系统的 3D 渲染成本观测（draw call、材质切换、阴影成本、后处理成本）

#### 4.0.3 资源与内容工作流层
- 更稳定的 Mesh / Material / Texture / Animation 资源导入与烹饪链路
- 更清晰的资源元数据、依赖追踪与失败诊断
- 更一致的逻辑路径资源模型在 3D 资源上的全面落地
- 更稳定的场景保存 / 加载 / 回归链路（面向 3D 场景，而非仅 2D 主线）
- 更明确的资源热重载与失效恢复策略

#### 4.0.4 编辑器工作流层
- 3D 组件 Inspector 的稳定收口
- 更可用的 3D Scene View 编辑体验（选中、移动、旋转、缩放、对齐、摄像机操作）
- 更稳定的 Play / Edit 切换与状态隔离
- 基础 3D 资源浏览、场景检视、调参闭环
- 面向 3D 最小工作流的编辑器验证链路，而不是“组件能显示就算支持”

#### 4.0.5 引擎能力层
- 更稳定的 3D 动画工作流（Animator / 状态机 / 骨骼动画链路收口）
- 更稳定的光照 / 阴影体系与参数调优能力
- 更明确的地形、粒子 3D、天空盒、摄像机等子系统的边界与优先级
- 更稳定的 3D 物理集成路径（当前 PhysX 相关构建链路仍不稳）
- 更清晰的 2D 主线与 3D 能力线隔离，避免总控持续被 3D 细节反向污染

#### 4.0.6 工程化与产品口径层
- 更明确的默认构建口径与 3D 开关策略
- 更稳定的 3D smoke / integration / regression 测试门禁
- 更一致的文档、样例、测试、构建表述
- 更克制的对外能力口径：先做到 MVP 可验证，再谈“现代化 3D 引擎”

换句话说，当前 3D 的问题不是“完全没有功能”，而是：

> 已经有若干能力点，但还缺少一个真正面向轻量级高性能现代化引擎的 **最小稳定闭环**。


## 4.1 目标定位

3D 版本当前不应定义为“完整 3D 引擎”，而应定义为：

**可验证、可演示、可回归的最小 3D 工作流版本。**

重点不是覆盖所有 3D 子系统，而是形成一条最小闭环链路：

- Mesh
- Camera
- Light
- Shadow
- Skybox
- 基础材质/资源加载
- 最小场景保存/加载
- 最小编辑器检视/运行验证

### 4.1.1 V0.10 - 3D MVP Internal

版本目标：完成内部可验证的 3D 最小闭环。

必须完成：

- `DSE_ENABLE_3D=ON` 时可稳定构建并运行基础样例
- 存在最小 3D 场景主链：Mesh + Camera + Directional Light + Shadow + Skybox
- 3D 基础资源路径使用统一逻辑路径模型
- 编辑器 Inspector 对关键 3D 组件可稳定检视
- 至少存在一条稳定的 3D smoke 测试链路
- 文档明确声明其为 MVP，不描述为默认稳定主线

建议门禁：

- 3D smoke 测试
- 最小 3D 场景启动验证
- 最小 3D 场景保存 / 加载验证

不纳入该版本：

- 全套地形生态
- 完整动画状态机编辑器
- 完整物理编辑工作流
- 完整后处理系统矩阵
- 完整商业级 3D 关卡制作链路

### 4.1.2 V0.11 - 3D MVP Preview

版本目标：从“内部可验证”升级为“外部可说明”的 3D MVP 预览版本。

必须完成：

- 3D 样例、测试、文档口径统一
- 基础 3D 场景在 Runtime 与 Editor 下均可验证
- Mesh / Animator / Particle3D / Terrain 中仅保留少量高价值稳定能力
- 3D 回归与构建开关路径更清晰

建议增加门禁：

- `engine.3d.*` 分组回归
- 3D 样例加载 smoke
- 关键资源链路失败可观测

不纳入该版本：

- 完整 3D 内容生产工具链
- 完整可视化材质系统
- 完整 GI / 光追 / 高级渲染体系
- AAA 级复杂场景承诺

### 4.1.3 3D MVP 对外口径

当 V0.11 达到后，建议对外统一表述：

- DSEngine 已具备最小可验证 3D MVP
- 可完成基础 3D 场景的运行、检视与回归
- 但当前默认稳定主线仍然是 2D
- 3D 仍处于收口阶段，不承诺完整商业级制作流

---

## 5. 推荐版本顺序

建议后续版本顺序固定为：

1. `V0.8 - 2D Stable Base`
2. `V0.9 - 2D Production Preview`
3. `V0.10 - 3D MVP Internal`
4. `V0.11 - 3D MVP Preview`

这样做的原因：

- 当前代码成熟度明显是 2D 高于 3D
- 编辑器结构问题会直接影响 2D 主线效率
- 如果 2D、Editor、测试门禁未稳，3D 会持续失去验证基础
- 先稳定主线，再收口 3D，整体成本最低

---

## 6. 每个版本的完成定义

### 6.1 V0.8 完成定义

满足以下条件即可认为 V0.8 完成：

- 2D 主链功能可稳定运行
- 关键 2D 模块回归通过
- 编辑器基础闭环可重复使用
- 主线文档与代码状态一致
- 构建脚本至少覆盖日常开发主路径

### 6.2 V0.9 完成定义

满足以下条件即可认为 V0.9 完成：

- 编辑器主入口已拆到可维护范围
- profiler、场景编辑、基础检视能力可持续使用
- 2D 项目具备持续迭代基础
- 测试与构建门禁开始形成稳定习惯

### 6.3 V0.10 完成定义

满足以下条件即可认为 V0.10 完成：

- 3D 最小样例可稳定跑通
- 3D 基础场景链路可保存 / 加载 / 回归
- 文档中明确标注其为 MVP Internal
- 不再把“有组件”误写成“有完整工作流”

### 6.4 V0.11 完成定义

满足以下条件即可认为 V0.11 完成：

- 3D 最小闭环可被外部理解与验证
- 3D 文档、样例、测试、构建口径一致
- 已有可重复运行的 MVP 级回归链路
- 仍保持对能力边界的克制表述

## 8. 总结

当前最合理的版本规划不是不断追加功能，而是明确两条线：

- 先完成 2D 稳定主线版本
- 再完成 3D 最小可验证 MVP 版本

如果按本文档推进，项目后续叙事会更清晰：

- 2D 是当前稳定主线
- Editor 是服务主线效率的工程化工具
- 3D 是按 MVP 标准逐步收口的方向

这比持续扩张功能面更符合当前仓库真实状态，也更有利于后续对外表达与内部执行。
